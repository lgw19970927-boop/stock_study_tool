## 功能一：CSV 匯出

### 1.1 需求摘要

在篩選結果統計列（stats-bar）已有「匯出 CSV」按鈕（UI 已建立，尚無 handler）。  
點擊後彈出 Modal 視窗，提供兩種匯出模式擇一，並讓使用者選擇儲存位置。

### 1.2 UI 示意圖

```
┌─────────────────────────────────────────────┐
│  匯出 CSV                              [✕]  │
├─────────────────────────────────────────────┤
│                                             │
│  請選擇匯出內容：                             │
│                                             │
│  ┌─────────────────────────────────────┐    │
│  │ ● 完整篩選結果                        │    │
│  │   含股票代碼、公司名稱、現價、漲跌幅、   │    │
│  │   成交量、篩選標籤等所有欄位             │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  ┌─────────────────────────────────────┐    │
│  │ ○ 僅匯出股票代碼 (Ticker)              │    │
│  │   僅包含篩選結果的股票代碼清單           │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  預覽：共 42 筆資料                          │
│                                             │
│          [ 取消 ]    [ 匯出並儲存 ]           │
│                                             │
└─────────────────────────────────────────────┘
```

### 1.3 技術方案

#### 前端（純瀏覽器端，不需後端 API）

| 步驟 | 說明 |
|------|------|
| 1 | 綁定 stats-bar 的「匯出 CSV」按鈕 click 事件 |
| 2 | 彈出 Modal 視窗（複用現有 Modal 樣式） |
| 3 | 從 `window.state.lastResults` 讀取篩選資料 |
| 4 | 根據使用者選擇（完整結果 / 僅 Ticker）組裝 CSV 字串 |
| 5 | 使用 [File System Access API](https://developer.mozilla.org/en-US/docs/Web/API/File_System_Access_API) 的 `showSaveFilePicker()` 選擇儲存位置 |
| 6 | fallback：若瀏覽器不支援（Firefox），使用 `<a download>` 方式直接下載 |

#### CSV 欄位定義

**完整篩選結果模式：**
```csv
Symbol,Name,Price,Change%,Volume,Matched_Indicators,Matched_Patterns
AAPL,Apple Inc.,195.50,+1.23%,45678900,MA金叉|RSI超賣,頭肩底
```

**僅 Ticker 模式：**
```csv
Symbol
AAPL
TSLA
NVDA
```

#### 檔案命名預設值
- 完整結果：`screening_results_YYYYMMDD_HHmmss.csv`
- 僅 Ticker：`screening_tickers_YYYYMMDD_HHmmss.csv`

### 1.4 邏輯細節

1. **無篩選結果時**：按鈕應為 disabled 狀態（灰色不可點擊），tooltip 提示「尚無篩選結果」
2. **篩選進行中**：按鈕同樣 disabled，避免匯出不完整資料
3. **CSV 編碼**：使用 UTF-8 with BOM（`\uFEFF` prefix），確保 Excel 開啟時中文不亂碼
4. **特殊字元處理**：公司名稱含逗號時用雙引號包裹（標準 CSV RFC 4180）

## 功能二：K 線圖顯示範圍切換

### 2.1 需求摘要

在 timeframe 按鈕列與「型態標註開關」之間，新增一個顯示範圍下拉選單。  
預設值為「1年：日K」，點擊後展開預設選項列表；列表最後有「+ 添加範圍」可開啟自訂範圍視窗。

#### 下拉選單（收合狀態）

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│ [1min][3min][5min][15min][30min][1H][4H][1D][1W][1M][1Y]  [1年:日K ▾]  [型態標註開關]  │
└──────────────────────────────────────────────────────────────────────────────────────┘
                                                              ↑ 新增元件
```

#### 下拉選單（展開狀態）

```
                              ┌──────────────┐
                              │ 1天：1分K     │
                              │ 5天：5分K     │
                              │ 10天：15分K   │
                              │ 20天：30分K   │
                              │ 3月：1小時K   │
                              │ 6月：日K      │
```

#### 「添加範圍」彈窗

```
┌───────────────────────────────────────────┐
│  添加範圍                            [✕]  │
├───────────────────────────────────────────┤
│                                           │
│  時間範圍：  [ 天 ▾ ]  [    ]  天          │
│                                           │
│  K線粒度：   [ ────── ▾ ]                  │

│                                           │
```

**時間範圍的單位下拉選項：** `天` / `月` / `年`  
**K線粒度下拉選項：** 依據所選「時間範圍單位」**動態過濾**（參考下方 §2.5 規則），避免出現不合理組合。

> **設計原則（參考東方財富「添加範圍」彈窗）：**  
> 1. K 線粒度選項**對齊現有 Timeframe 按鈕列**（11 項），不額外新增 10分K、2小時K、3小時K、季K 等後端尚未支援的粒度。  
> 2. 根據使用者選擇的「時間範圍單位」，K 線粒度下拉**即時隱藏不合理的選項**（而非等使用者選完再驗證報錯），體驗更流暢。

|----------|---------|---------------|-----------------|------|
| 1天：1分K | 1 天 | 1 分鐘 | `1m` | 日內短線 |
| 5天：5分K | 5 天 | 5 分鐘 | `5m` | |
| 10天：15分K | 10 天 | 15 分鐘 | `15m` | |
| 20天：30分K | 20 天 | 30 分鐘 | `30m` | |
| 3月：1小時K | 3 個月 | 1 小時 | `1h` | |
| 6月：日K | 6 個月 | 日 | `1d` | |
| **1年：日K** | **1 年** | **日** | **`1d`** | **預設值** |
| 3年：日K | 3 年 | 日 | `1d` | |
| 10年：日K | 10 年 | 日 | `1d` | |
| 全部：周K | 全部歷史 | 週 | `1w` | |

### 2.4 核心邏輯：顯示範圍 ↔ Timeframe 按鈕 互動規則

#### 規則 1：從顯示範圍下拉選單選擇 → 同步更新 Timeframe 按鈕

> 選擇「3年：日K」→ Timeframe 按鈕 `1D` 高亮，圖表載入 3 年日 K 數據

```
使用者選擇顯示範圍預設值
  → 解析出 { rangeDuration, timeframe }
  → 更新 Timeframe 按鈕 UI（高亮對應按鈕）
  → 呼叫 loadStock() 並設定 visibleRange 為 rangeDuration
```

#### 規則 2：從 Timeframe 按鈕切換 → 顯示範圍下拉標籤同步更新

> 維持目前專案邏輯（切 timeframe → 重新 fetch → 顯示最近一年）  
> 下拉選單標籤更新為「1年：{新timeframe名稱}」

```
使用者點擊 Timeframe 按鈕（如 1W）
  → 維持現有邏輯：loadStock() with period='max', 然後 setVisibleRangeToLastYear()
  → 更新下拉選單顯示文字為「1年：周K」
  → 若該組合不在預設列表中，標籤顯示自訂文字但不添加進列表
```

#### 規則 3：自訂添加範圍 → 添加至下拉選單底部（「+添加範圍」之前）

```
使用者透過添加範圍視窗確定
  → 驗證組合合理性（見 2.5）
  → 新預設項目插入下拉選單
  → 立即套用該範圍
  → 自訂項目儲存於 localStorage 以跨 session 保留
```

#### 規則 4：切換股票時 → 保持當前選擇的顯示範圍

> 載入新股票時，沿用當前 dropdown 選擇的 {rangeDuration, timeframe}，不重置為預設。

#### 規則 5：`fromFilterClick` 模式 → 跟隨篩選分析結束日期

> 從篩選結果點擊股票時，visible range 結束日仍對齊 `analysis_end_date`（現有邏輯），  
> 但範圍長度改為依照當前下拉選單的 rangeDuration（而非固定 1 年）。

### 2.5 合理性驗證 & 限制

> **設計參考：** 東方財富的「添加範圍」彈窗會根據選擇的時間範圍單位（天/月/年），  
> 動態過濾 K 線粒度下拉選項——例如選「年」時只保留日K以上的粗粒度。  
> 本系統採用相同做法（兩層防護：動態過濾 + 數值上限驗證）。

#### 第一層：依時間範圍單位動態過濾 K 線粒度選項

使用者選擇「時間範圍」的單位後，K 線粒度下拉**立即只顯示合理的選項**：

| 時間範圍單位 | 可選 K 線粒度 | 隱藏的選項 | 參考依據 |
|------------|-------------|-----------|----------|
| **天** | 全部 11 項 | （無） | 天級範圍配所有粒度皆合理 |
| **月** | 3分K、5分K、15分K、30分K、1小時K、4小時K、日K、周K、月K、年K（10 項） | 1分K | 多月 × 1分鐘資料量過大（參考截圖 2） |
| **年** | 日K、周K、月K、年K（4 項） | 所有日內粒度 | 年級範圍僅需日線以上粗粒度（參考截圖 3） |

> **與參考 App 的差異：** 參考 App 在「年」範圍不顯示周K，但周K 配年級範圍完全合理（1 年 ≈ 52 根 K 棒），故本系統保留。

#### 第二層：數值上限驗證（已過濾粒度內的額外防護）

即使粒度已通過第一層過濾，仍需限制時間跨度數值避免資料量爆炸：

| K 線粒度 | 允許的最大時間跨度 | 原因 |
|---------|------------------|------|
| 3分K / 5分K | ≤ 30 天（天）/ ≤ 1 月（月） | 分鐘級資料量大 |
| 15分K / 30分K | ≤ 90 天（天）/ ≤ 3 月（月） | 同上 |
| 1小時K / 4小時K | ≤ 365 天（天）/ ≤ 12 月（月） | 小時級資料 |
| 日K | ≤ 30 年 | 日線資料合理上限 |
| 周K / 月K / 年K | 無限制 | 粗粒度可支援全歷史 |

**不合理組合的處理方式：**
1. **第一層（過濾）：** 切換時間範圍單位時，K 線粒度下拉即時重新渲染可用選項。若當前選中的粒度被隱藏，自動回落到過濾後的第一個選項。
2. **第二層（驗證）：** 數字輸入框根據當前粒度動態設定 `max` 屬性。超出上限時確定按鈕灰色不可點擊並顯示提示文字。

### 2.6 後端影響

| 項目 | 是否需要修改 | 說明 |
|------|------------|------|
| 市場數據 API | **不需要** | 現有 `/api/market-data/{symbol}?interval=&period=max` 已支援所有 timeframe，前端自行控制 `setVisibleRange()` |
| 重採樣支援 | **不需要** | K 線粒度選項已對齊 Timeframe 按鈕列，所有粒度均在 RESAMPLE_CONFIG 中已有支援 |
| 新增可能的 API 優化 (P2) | 選擇性 | 當顯示範圍較短（如 1 天 / 5 天）時，不必 `period=max` 取全部數據。可改為 `start_date` + `end_date` 精確取值，減少傳輸量與查詢時間 |

#### 後端 RESAMPLE_CONFIG

**無需修改。** K 線粒度選項已對齊現有 Timeframe 按鈕，所有粒度（`1m`, `3m`, `5m`, `15m`, `30m`, `1h`, `4h`, `1d`, `1w`, `1M`, `1y`）在 RESAMPLE_CONFIG 中均已有對應。

```javascript
function rangeDurationToSeconds({ value, unit }) {
    const multipliers = {

        year:  365 * 24 * 60 * 60,
    };

}
```

## 隱藏邏輯 & 待確認問題

### ✅ 問題 1：自訂範圍刪除機制（已確認）

自訂項目旁顯示 `🗑` 刪除按鈕（常駐顯示），預設項目不可刪除。  
刪除時同步移除 localStorage 中的對應記錄。若刪除的是當前選中的範圍，自動回落到預設值「1年：日K」。

### ✅ 問題 2 & 3 & 5：K 線粒度選項範圍（已確認）

**K 線粒度下拉選項嚴格對齊現有 Timeframe 按鈕列**，即：  
`1分K` / `3分K` / `5分K` / `15分K` / `30分K` / `1小時K` / `4小時K` / `日K` / `周K` / `月K` / `年K`

不包含 `10分K`、`2小時K`、`3小時K`、`季K`。  
理由：這些粒度在後端 `RESAMPLE_CONFIG` 中尚未支援，前端不應提供後端無法回傳資料的選項。  
未來若後端補齊，再同步開放。

> **MVP (Minimum Viable Product，最小可行產品)**：指功能的第一個可用版本，只包含核心必要邏輯，不包含進階/邊緣功能。目標是以最小開發成本驗證功能可用性。

**後端影響：無需修改 RESAMPLE_CONFIG。** 所有選項均已有後端支援。

### ✅ 問題 4：「全部：周K」的定義（已確認）

「全部」= 資料庫中該股票最早到最新的所有數據。  
實作：API 使用 `period=max`，前端呼叫 `chart.timeScale().fitContent()` 自動填滿。

### ✅ 問題 6：顯示範圍與篩選分析時間範圍的關係（已確認）

兩者完全獨立。顯示範圍僅影響 K 線圖可見區間，不影響篩選結果。

### ✅ 問題 7：CSV 匯出內容（已確認）

CSV 只匯出文字標籤（`matched_indicators`、`matched_patterns`），不包含指標計算數值。  
無需修改後端 SSE 回傳格式。

## 風險評估

### 功能一（CSV 匯出）風險：低

| 風險 | 嚴重度 | 說明 | 緩解措施 |
|------|-------|------|---------|
| `showSaveFilePicker` 在 Firefox 不可用 | 低 | Firefox 不支援 File System Access API | 已規劃 `<a download>` fallback，自動觸發下載到預設下載資料夾 |
| 大量資料效能 | 極低 | 美股篩選最多 ~8000 支，CSV 字串生成為毫秒級 | 無需擔心 |

**什麼情況會失敗：**  
- 幾乎不會。唯一邊界情況是使用者在篩選執行中點匯出（已由 disabled 狀態防護）。

### 功能二（顯示範圍切換）風險：中

| 風險 | 嚴重度 | 說明 | 緩解措施 |
|------|-------|------|---------|
| Timeframe 按鈕 ↔ 顯示範圍雙向同步 bug | 中 | 兩個控制項互相影響，容易產生循環觸發 | 使用 `_isUpdatingFromDropdown` / `_isUpdatingFromButton` 鎖避免循環 |
| 分鐘級資料不存在 | 中 | 使用者選「1天：1分K」但該股沒有 1m 數據 | API 回傳空資料時顯示提示「此股票無 1 分鐘級數據」 |

| 自訂範圍存 localStorage 讀取失敗 | 極低 | JSON parse 失敗 | try-catch fallback 到空陣列 |

**最大風險：** 雙向同步邏輯（Timeframe 按鈕 ↔ 顯示範圍下拉選單）。  
**什麼情況會失敗：** 如果同步鎖機制有 race condition，可能導致 UI 狀態不一致（標籤顯示內容與實際圖表不符）。  
**替代方案：** 將 Timeframe 按鈕改為唯讀指示器（不可點擊），所有 timeframe 切換都通過顯示範圍下拉完成。但這會大幅改變現有使用者體驗，不建議。
