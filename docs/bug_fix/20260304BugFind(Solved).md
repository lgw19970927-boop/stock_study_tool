# 2026-03-04 指標篩選效能優化計畫

## 背景
將系統從 SQLite 升級為 MySQL 架構後，即使是簡單的「指標篩選」，執行速度也變得無法忍受（超過 10 秒以上），連帶導致沒有資料或已下市的股票也拖垮整體系統。
根本原因在於原 SQLite 架構的「歷史共業」被 MySQL 的 TCP/IP 網路延遲給放大了。

### 原 SQLite 隱藏的三大效能缺陷：
1. **未實作 Active 過濾**：篩選會撈出全市場股票（包含 89 檔甚至數千檔下市、停牌股票），送進後續進行毫無意義的 K 線查詢與 Pandas 運算。
2. **無防呆機制的重度運算（停牌容忍度未實作）**：原計畫書提到的「大盤基準日」機制並未在指標篩選中實作，導致只要有 K 線，哪怕是 3 年前停牌的股票，都會被抓出來算一輪。
3. **無差別全歷史拉取 (最大元兇)**：現有架構在查 K 線時**完全沒設定日期限制**（`SELECT * FROM market_data_ohlcv WHERE symbol=?`）。例如 AAPL 會把 10 年內數千筆資料透過 MySQL 網路拉進來轉 Pandas DataFrame，然後才在記憶體中把不要年份的資料切掉，浪費極大的資料傳輸與記憶體建置時間。

---

## 預計實作之優化方案

### 1. 落實 `Active` 過濾（第一線攔截）
- **實作位置**：`App/Feature/Screening/routes.py`
- **作法**：在 `filter_stocks_stream` 的 SQL 查詢中，加入 `AND status = 'Active'`。
- **效益**：第一關就擋掉因下市或代碼變更而報廢的無效實體，減少分母。

### 2. 實作停牌容忍度機制（大盤基準日快篩）
- **實作位置**：`App/Feature/Screening/service.py` (`screen_stocks`)
- **作法**：
  - 每次篩選啟動時，先拉取指標股（例如 `SPY` 或 `QQQ`）最新一筆日 K 線日期作為大盤基準日。
  - 此機制會在篩選前先行過濾（或在逐筆查詢時動態驗證），確保距離大盤最後交易日超過一定容忍天數（例如 3~5 天）的股票直接棄權，不載入全量資料、不傳入 Pandas。

### 3. 動態歷史拉取擷取（解決全量載波問題）
- **實作位置**：`App/Feature/Screening/service.py` (`screen_single_stock` & Indicator 解析)
- **作法**：
  - 在開始拉取 K 線前，先解析出前端要求的**最大需要天數 `req_period`**（例如：最長需要 MA60，就是 60 根 K 線）。
  - 利用該週期推算出一個「安全擷取起始日」：大約是 `(Max_Period) * 1.5 + 30` 天前。
  - 將核心 SQL 查詢改為：`SELECT ... WHERE datetime >= %s`。
- **效益**：把過濾工作交給強大的 MySQL 引擎，只把「算近期指標真正需要的少數 K 棒」拉回後端，徹底消除網路 I/O 瓶頸，讓 Pandas 的計算負載降至最低。

---

## 驗證指標
完成這三項修改後，13,000 檔股票的篩選時間應恢復甚至超越原本 SQLite 檔案 I/O 的速度，且能從後端 Logger 確認被過濾掉的下市或停牌標的。

## 補充：前端狀態保留機制修復

除了效能優化外，也修復了前端在切換分頁時參數會遺失的關鍵問題。

### 背景
由於系統前端架構使用 HTMX 進行分頁內容的動態抽換，當使用者由「股票篩選」切換到其他分頁（如「模擬交易回測」）再切回時，原本填寫在畫面上的 DOM 元素（輸入框、指標卡片、Checkbox 等）會被完整銷毀並重新載入預設的空白結構。

### 實際修復方案
1. **確立全域狀態 (State) 中心**
   - 所有的參數異動，在發生時都會自動儲存在全域變數 `window.state.filters` 中。
2. **實作狀態反填機制 (`syncFromState`)**
   - 在各個核心篩選區塊（`market_block.js`, `time_range_block.js`, `pattern_block.js`, `indicator_block.js`）新增 `syncFromState` 方法。
   - 每次 HTMX 重新 `afterSettle` 觸發初始化 `init()` 時，區塊會自動去讀取 `window.state.filters`，並將儲存的選取狀態「重新渲染並填寫」回全新的 DOM 上。
3. **解決 HTMX 腳本重複宣告的致命錯誤**
   - 修復了 `config.js` 與 `app.js` 等全域腳本因 HTMX 重新執行 `<script>` 而觸發 `const` / `function` 變數重複宣告 (SyntaxError) 的問題。將寫法改為 `var` 宣告或掛載在 `window.xxx` 下，確保再次切回分頁時，狀態還原函式能被順利執行而不會崩潰。 
4. **指標動態重建**
   - 對於動態生成的「指標篩選卡片」（如 MA, BOLL），實作了深度還原邏輯：在 `syncFromState` 過程中，會根據全域紀錄動態呼叫 `restoreState()` 與 `confirmConfig()`，一比一重建整個參數卡片。
