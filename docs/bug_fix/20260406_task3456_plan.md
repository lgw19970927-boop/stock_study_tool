### 3.3 BOLL 升穿上軌等四個 Preset

**後端確認**：⚠️ **Crossover 邏輯未實作，存在語意 Bug**

#### 現況（Bug）

前端生成的 condition（以升穿上軌為例）：
```json
{ "left": "close", "operator": ">", "right": "BB_UPPER" }
```

後端以 `evaluate_condition()` 直接判斷**當根** `close > BB_UPPER`，這只是「收盤在上軌之上」，不是「升穿（前根不在上軌上方、當根突破）」。

#### 修復方案

**「當前值」模式**（range = 當前值）：
真正 crossover 邏輯：
- 升穿上軌：`prev_close <= BB_UPPER_prev` AND `current_close > BB_UPPER_current`
- 跌穿下軌：`prev_close >= BB_LOWER_prev` AND `current_close < BB_LOWER_current`
- 升穿中軌、跌穿中軌：同理

後端在處理 preset 類型時，需對 `eval_df` 取最後兩根特殊判斷：

```python
BOLL_CROSSOVER_PRESETS = {
    '升穿上軌':   ('BB_UPPER',  '>'),
    '升穿中軌':   ('BB_MIDDLE', '>'),
    '跌穿中軌':   ('BB_MIDDLE', '<'),
    '跌穿下軌':   ('BB_LOWER',  '<'),
}

if preset and preset in BOLL_CROSSOVER_PRESETS and not is_consecutive:
    band_col, direction = BOLL_CROSSOVER_PRESETS[preset]
    if len(eval_df) < 2:
        indicator_met = False
    else:
        cur_close = eval_df.iloc[-1]['close']
        cur_band  = eval_df.iloc[-1][band_col]
        pre_close = eval_df.iloc[-2]['close']
        pre_band  = eval_df.iloc[-2][band_col]
        if direction == '>':
            indicator_met = (pre_close <= pre_band) and (cur_close > cur_band)
        else:
            indicator_met = (pre_close >= pre_band) and (cur_close < cur_band)
```

**「連續週期 N」模式（用戶確認的語義設計）**：

語意：**「N 根前發生升穿，此後 N 根均維持在軌道同側」**，代表「突破後站穩」的交易訊號。

具體判斷：
- `eval_df.iloc[-(N+1)]`（window 前一根）：`close <= BB_UPPER`
- `eval_df.iloc[-N]` 至 `eval_df.iloc[-1]`（共 N 根）：全部 `close > BB_UPPER`

```python
if preset and preset in BOLL_CROSSOVER_PRESETS and is_consecutive:
    band_col, direction = BOLL_CROSSOVER_PRESETS[preset]
    # 需要 N+1 根資料（window + 前一根）
    if len(eval_df) < range_n + 1:
        insufficient.append(...)
        continue
    pre_bar  = eval_df.iloc[-(range_n + 1)]
    window   = eval_df.iloc[-range_n:]

    if direction == '>':
        # 前一根未突破 + 後 N 根均在上方
        breakout_valid = pre_bar['close'] <= pre_bar[band_col]
        sustained = all(row['close'] > row[band_col] for _, row in window.iterrows())
    else:
        breakout_valid = pre_bar['close'] >= pre_bar[band_col]
        sustained = all(row['close'] < row[band_col] for _, row in window.iterrows())

    indicator_met = breakout_valid and sustained
```

#### 連續週期設計的邏輯合理性說明（⚠️ 邏輯待確認，實作前請先回顧此節）

| 方案 | 語意 | 評估 |
|------|------|------|
| **A（現行計畫）** | 第 1 根升穿，後 N-1 根站穩上軌上方，共 N 根 | ✅ 「突破後站穩」確認型訊號 |
| B（用戶提議） | 前 N-1 根維持在上軌下方，最後 1 根升穿 | ⚠️ 「壓縮後剛突破」捕獲型訊號；當根即訊號 |
| C | 所有 N 根均在上軌上方（不驗證突破時間點） | ❌ 無法排除「早已突破的舊趨勢」，不等同升穿 |

> **「連續N次 + 自訂 UPPER < 價格」≠ Plan A**  
> `連續N次 + UPPER < 價格` 只檢查 N 根均在上軌上方，不驗證 window 前一根是否在上軌下方。  
> Plan A 額外要求 `eval_df.iloc[-(N+1)]` 在上軌以下，確保這是「新突破」而非舊趨勢延續。

**暫採方案 A**（討論確認中，邏輯部分暫不實作，等確認後再開發）。

## 任務四：user_data 備份時機修訂 & 策略即時持久化

### 4.1 現況

| 項目 | 現況 |
|------|------|
| 策略儲存 | 僅存於 `window.state.savedStrategies`（in-memory，重整即消失）|
| `user_data.strategies` | 表結構完整，但無任何 API 寫入，實為空表 |
| `backup_user_data()` | 正常可用，但備份的是空表，無實際意義 |
| 觸發時機 | 每日 23:55 固定排程（`data_sync` container 的 `scheduler.py`）|

### 4.2 目標流程

```
使用者在「我的策略」Tab 執行 儲存 / 更新 / 刪除
          ↓
前端 StrategyManager 呼叫 REST API
          ↓
FastAPI 後端寫入 MySQL user_data.strategies
          ↓
觸發備份（見 4.4 架構說明）
          ↓
覆寫 env/mysql/seed/seed_user_data.sql
```

### 4.3 備份內容

`seed_user_data.sql` 備份整個 `user_data` schema，包含：

| 資料表 | 備份內容 | 本版本是否有資料 |
|--------|--------|------------|
| `strategies` | 策略名稱、configuration（含所有篩選指標 JSON）、is_active、timestamps | ✅ 本版本實作後開始有資料 |
| `screening_results` | 歷史篩選結果快取 | ❌ 目前無資料，但 schema 一同備份 |

### 4.4 備份架構問題與解決方案

**關鍵限制**：`backup_user_data()` 使用 `docker exec stock-mysql mysqldump`，此指令需在有 Docker socket 的 container 中執行。查看 `docker-compose.yml`：

| Container | Docker socket 掛載 |
|-----------|------------------|
| `data_sync` | ✅ `/var/run/docker.sock:/var/run/docker.sock` |
| `fastapi` | ❌ 未掛載 |

**因此 FastAPI container 無法直接呼叫現有 `backup_user_data()`。**

#### 推薦方案：資料寫入 FastAPI，備份交由 data_sync 排程（加密排程）

```
fastapi ──寫入──→ MySQL user_data.strategies ←──data_sync scheduler（每 15 分鐘）──→ mysqldump → SQL
```

- FastAPI 只負責 CRUD 寫入 DB，不負責備份
- `data_sync/scheduler.py` 新增 `user_data` 備份排程，從每日 23:55 改為**每 15 分鐘**
- 因資料極小（~4KB），頻繁備份無 I/O 負擔
- 最壞情境：使用者儲存策略後 14 分鐘內關機 → 最多丟失 14 分鐘內的操作

修改 `scheduler.py`：
```python
# 舊排程（每日 23:55）移除或保留作最後防線
# 新排程（每 15 分鐘）
scheduler.add_job(
    backup_user_data,
    'interval',
    minutes=15,
    id='backup_user_data_frequent',
    name='User Data Backup (every 15 min)',
    coalesce=True,
    max_instances=1,
)
```

#### 備選方案：FastAPI 實作純 Python 備份（無需 docker exec）

在 FastAPI container 中，由於已有 MySQL 連線憑證，可用 Python 直接導出：
- 利用 `pymysql` 讀取 `user_data.strategies` 並生成 SQL INSERT 語句
- 寫入相同的 `seed_user_data.sql` 路徑
- 無需 Docker socket、無需 mysqldump binary
- 缺點：手寫 SQL dump 不如 `mysqldump` 完整（缺少 CREATE TABLE 等 schema 部分）

> **建議**：先採用「加密排程（每 15 分鐘）」方案，實作最簡單；若業務需要即時備份再改為備選方案。

## 任務五：篩選列表漲跌幅 +0% 顯示修正

### 現況問題

`results_table.js` 中判斷：
```js
const changeToneClass = stock.change_percent >= 0
    ? 'text-color-success bg-[rgba(63,185,80,0.1)]'   // 0% 也會變綠色 ❌
    : 'text-color-danger  bg-[rgba(248,81,73,0.1)]';
const changeSign = stock.change_percent >= 0 ? '+' : '';  // 0% 顯示 +0% ❌
```

當 `change_percent === 0` 時顯示 `+0%`，背景綠色，視覺誤導為上漲。

### 修復方案

```js
const isZero = stock.change_percent === 0;
const changeToneClass = isZero
    ? 'text-text-secondary bg-[rgba(150,150,150,0.1)]'   // 灰色
    : stock.change_percent > 0
        ? 'text-color-success bg-[rgba(63,185,80,0.1)]'  // 綠色
        : 'text-color-danger  bg-[rgba(248,81,73,0.1)]'; // 紅色
const changeSign = stock.change_percent > 0 ? '+' : '';   // 0% 不顯示 +
```

顯示效果：
- `+2.5%` → 綠色背景，有 `+` 號
- `-1.3%` → 紅色背景，有 `-` 號（原本 `change_percent` 已含負號）
- `0%` → 灰色背景，無前置符號

**修改檔案**：`app/feature/screening/components/results_table/results_table.js`

```js
document.addEventListener('keydown', (e) => {
    // 焦點守衛：輸入框有焦點時不觸發，避免干擾策略名稱等欄位
    const tag = document.activeElement.tagName;
    if (['INPUT', 'SELECT', 'TEXTAREA'].includes(tag)) return;

    if (e.key !== 'ArrowUp' && e.key !== 'ArrowDown') return;

    const items = [...document.querySelectorAll('#stockList .stock-item')];
    if (!items.length) return;

    e.preventDefault();  // 避免頁面捲動

    // 未選中任何 row 時，方向鍵不做任何事
    const currentIdx = window.ScreeningPage._selectedStockIndex;
    if (currentIdx === null) return;

    let newIdx = currentIdx;
    if (e.key === 'ArrowDown') newIdx = Math.min(currentIdx + 1, items.length - 1);
    if (e.key === 'ArrowUp')   newIdx = Math.max(currentIdx - 1, 0);

    window.ScreeningPage._selectedStockIndex = newIdx;

    // 更新反白
    items.forEach((el, i) => el.classList.toggle('stock-item--selected', i === newIdx));

    // 確保選中項目進入視野（不影響左側 sidebar 捲動）
    items[newIdx].scrollIntoView({ block: 'nearest', behavior: 'smooth' });

    // 載入 K 線圖
    const symbol = items[newIdx].querySelector('.stock-symbol')?.textContent?.trim();
    if (symbol) window.ScreeningPage.onStockClick(symbol);
});
```

### A.3 篩選器策略與資料庫的連動說明

**篩選 Tab 的「執行篩選」不寫入資料庫**；它是即時計算，結果只顯示在 UI，不持久化。

**「我的策略」Tab 的操作（儲存 / 更新 / 刪除）全部與資料庫連動**，透過 REST API：
- `POST /api/strategies` → 寫入 `user_data.strategies`
- `PUT /api/strategies/{id}` → 更新 `user_data.strategies`
- `DELETE /api/strategies/{id}` → 軟刪除（`is_active=0`）

頁面重新整理後，「我的策略」Tab 呼叫 `GET /api/strategies` 從資料庫重新載入，策略會持久保留。

### A.4 Volume（mysql_data）vs env/mysql 的角色說明

| 項目 | 角色 | 位置 |
|---|---|---|
| **Volume `mysql_data`** | MySQL 實際的資料存放位置（Docker named volume）| `/var/lib/mysql`（container 內）|
| **`env/mysql/seed/seed_user_data.sql`** | 備份/還原用的 SQL dump 檔案（git 可追蹤）| `d:\Projects\stock_study_tool\env\mysql\seed\` |
| **`env/mysql/seed/seed_market_data.sql`** | 行情資料的 SQL dump | 同上 |

**Volume `mysql_data` 是唯一的 live 資料來源**。`env/mysql/seed/*.sql` 是快照備份，僅在以下兩種情境被讀取：
1. MySQL container **首次初始化**（`/docker-entrypoint-initdb.d/` 掛載機制，只跑一次）
2. **手動重建 volume**（`docker volume rm mysql_data`）時還原資料

### A.5 備份觸發時機

| 備份類型 | 觸發者 | 時機 | 目標檔案 |
|---|---|---|---|
| `user_data` 備份 | `data_sync` container APScheduler | **每 15 分鐘**（`interval minutes=15`）| `env/mysql/seed/seed_user_data.sql` |
| `market_data` 備份 | `data_sync` container APScheduler | **每日 23:59**（`cron hour=23 minute=59`）| `env/mysql/seed/seed_market_data.sql` |
| MySQL Volume | Docker named volume | **隨時同步**（任何 SQL 寫入立即持久化）| Docker volume `mysql_data` |

> **結論**：Volume `mysql_data` 的資料是即時的（任何 CRUD 立即生效）；`env/mysql/seed/*.sql` 是定期快照，`user_data` 最多落後 14 分鐘，`market_data` 最多落後 23 小時 59 分鐘。

目前**不支援「Volume 有修改就立即備份 seed SQL」**，因為：
1. FastAPI container 無 Docker socket，無法執行 `mysqldump`
2. 即時備份（每次 API 寫入後觸發）會大幅增加 I/O 且難以處理並發
3. `user_data` 體積極小（~4KB），每 15 分鐘備份已足夠

如需改為「有修改立即備份」，建議方案是在 FastAPI 中用 pymysql 直接生成 INSERT SQL（無需 mysqldump），並在每次 strategy CRUD 成功後非同步寫入檔案。

## 附錄 B：BOLL Preset 顯示規則修改建議

### B.1 背景：BOLL Preset 已支援自訂參數

任務三實作後，BOLL 四個 Preset（升穿上軌 / 升穿中軌 / 跌穿中軌 / 跌穿下軌）改為**可自訂計算週期（p）與標準差（std）**，不再固定為 BOLL(20,2)。

因此，若兩個 Preset 條件同為「升穿上軌」但 parameters 不同（例如 p=20,std=2 與 p=10,std=1.5），目前顯示均為 **`BOLL-日K: 升穿上軌`**，無法區分。

### B.2 現況顯示規則（實作後）

**摘要模式**（指標卡片 summary line，見 `bollinger.js` → `confirmConfig()` + `format_helpers.js`）：

| 模式 | 範例顯示 |
|---|---|
| Preset，N=1 | `BOLL-日K: 升穿上軌(20,2)` |
| Preset，連續 N=3 | `BOLL-連續3次日K: 升穿上軌(20,2)` |
| Custom | `BOLL-日K: UPPER20_2>價格` |

**篩選標籤**（結果列表 tag，見 `service.py` → `build_tag()`）：

| 模式 | 範例顯示 |
|---|---|
| Preset，N=1 | `1D: BOLL 升穿上軌(20,2)` |
| Preset，連續 N=3 | `3D: BOLL 升穿上軌(20,2)` |
| Custom | `1D: BOLL 價格>UPPER20_2` |

### B.3 修改建議

#### 方案 1：僅在非預設參數時加括號（不採用）

若僅在「非預設值」才顯示參數，會讓使用者難以一眼判斷目前策略的 BOLL 參數。

#### 方案 2：一律顯示參數（已採用）

```
BOLL-日K: 升穿上軌(20,2)    ← 始終顯示
BOLL-日K: 升穿上軌(10,1.5)  ← 自訂參數同樣顯示
1D: BOLL 升穿上軌(20,2)
1D: BOLL 升穿上軌(10,1.5)
```

#### 實作決策：採用方案 2

| 考量 | 方案 1 | 方案 2 |
|---|---|---|
| 可讀性 | ❌ 需額外判斷「是否預設」 | ✅ 一眼看出實際參數 |
| 一致性 | ❌ 同一 Preset 可能有兩種顯示 | ✅ 所有 Preset 同格式 |
| 實作複雜度 | 中（需判斷是否等於預設值）| 低（一律附加） |

### B.4 實作位置

| 層面 | 修改位置 | 說明 |
|---|---|---|
| 摘要模式（前端）| `bollinger.js` → `confirmConfig()` 中 `displayConditions.push(preset.name)` | 改為 `preset.name + paramSuffix(pVal, stdVal)` |
| 篩選標籤（後端）| `service.py` → Preset 分支中 `preset_conditions` 的字串建法 | 改為 `f"{preset}({p},{std_str})"`（一律附加） |
| 策略卡片摘要（前端）| `strategy_manager.js` → `_buildIndicatorSummaryLines()` Preset 分支 | 讀取 `indicator.parameters` 並一律加後綴 |

#### 前端 helper 建議（`bollinger.js` confirmConfig）：
```js
function _presetParamSuffix(p, std) {
    const stdStr = (std % 1 === 0) ? String(parseInt(std)) : String(std);
    return `(${p},${stdStr})`;
}
// 使用：
displayConditions.push(`${preset.name}${_presetParamSuffix(pVal, stdVal)}`);
```

#### 後端建議（`service.py` → `_format_preset_name_with_params`）：
```python
DEFAULT_BOLL_P, DEFAULT_BOLL_STD = 20, 2.0

def _format_preset_name(preset_name: str, params: dict) -> str:
    p   = int(params.get("period", params.get("p", DEFAULT_BOLL_P)))
    std = float(params.get("std_dev", params.get("std", DEFAULT_BOLL_STD)))
    std_str = str(int(std)) if std == int(std) else str(std)
    return f"{preset_name}({p},{std_str})"
```

> **注意**：`strategy_manager.js` 的策略卡片摘要也需同步修改，否則從資料庫恢復的策略顯示會與即時設定不一致。
