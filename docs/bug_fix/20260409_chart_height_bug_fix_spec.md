# 圖表高度 Bug 修復規格與紀錄 (Bug 1, Bug 2, Bug 3)

**更新日期**：2026-04-09
**相關文件**：`docs/bug_fix/20260408_chart_height_spec.md`

## 發現的問題

透過本機測試與分析程式碼，我們確認了使用者回報的三個異常現象及其根本原因：

### 問題一：單一主圖時，拖拉整體高度導致主圖被異常壓縮（內部留白）[Not fixed,still have issue]
* **現象**：在沒有開啟任何副圖時，按住圖表區最下方的「拖拉調整整體圖表高度」控制把手往上拖拉。當整體容器縮小到了「最小高度限制」而停止縮小後，如果您繼續把滑鼠往上拉，**主圖的內部圖表竟然還會繼續被壓縮變小**，導致圖表區內部下方出現一大塊異常的黑底留白。拖拉把手不應該在容器停止縮放後，還能單獨去改變內部 LightweightCharts 容器的縮放行為。
  * **根本原因**：在 `chart_controller.js` 的 `setChartHeightByDrag` 方法中，計算拖拉距離時分為了 `nextHeight`（無下限的滑鼠拖拉計算值）與 `constrainedNextHeight`（受限於容器最小高度的實際高度）。但在處理單一主圖的情境時，錯誤地使用了未受限制的 `nextHeight` 傳給 `panes[0].setHeight(mainH)`。這導致當容器本身因為下限而停止縮小的時候，內部的主圖仍依照滑鼠的負值 `nextHeight` 繼續被強制縮小 (被壓到只剩 `_minMainPaneHeight`)，造成外層容器大、內部畫布小的「脫鉤留白現象」。
* **預期效果**：在只有主圖的情境下，不論將底部把手往上推至最小限制還是往下拉大，主圖的 K 線與時間軸都應該完美貼合容器尺寸。推到最小高度極限繼續往上拉時，主圖不會異常變小，也不會露出黑色留白區域。

### 問題二：新增或變更副圖數量時，錯誤壓縮既有副圖，關閉時甚至引發殘存副圖暴漲[Fixed]
* **現象**：
  1. 如果同時開啟 VOL 跟 RSI（或任何順序開啟多個），會發現**越早被 LightweightCharts 掛載的副圖越容易遭到嚴重壓縮**（變成極窄的長條）。此現象即便在**手動拖拉底部把手放大整體容器高度後**，仍然會發生，不受容器大小影響。
  2. 若在「VOL 處於嚴重壓縮」的畸形狀態下，直接將 VOL 關閉（點擊 X），空出來的空間**不僅沒有還給主圖**，反而在畫面上唯一剩下的 **RSI 會異常暴漲到佔據幾乎整個版面**，把主圖往上死角擠壓。若事先拉大整體圖表的高度，此「暴漲」現象會變得更加嚴重。
* **根本原因**：
  1. **LW 均分的不可控性**：當圖表同時創建多個副圖，或是接連掛載 Series 時，LightweightCharts 會自動對所有的 pane 進行預設均分或重構。若我們未在創建後強硬執行自己的比例，先建立的 Pane 就會被後建立的 Pane 自動擠壓。
  2. **致命的 `scaleFactor` 錯位計算**：在 `_captureCurrentPaneHeights`（於每次增減副圖前執行）中，程式碼試圖用 `panes.slice(0, expectedPaneCount)` 取出總和來當作分母算 `scaleFactor`。當您關閉 VOL 剩下 RSI 時，`expectedPaneCount` 變成 2 (主圖 + 1副圖)。因此它拿了 `panes[0]` (主圖) 與 `panes[1]` (即將被關掉的萎縮 VOL) 的高度加總來當分母，得出了極小的值。總容器高度除以這個極小分母，產生了 **極度巨大（如 7 到十幾倍不等）的 `scaleFactor`**（若容器預先被拉大，此倍數會更誇張）。隨後，程式又將此巨大的倍率乘上了 `RSI` 當時的高度，寫入了 `state.RSI.savedHeight`，導致 RSI 下一秒渲染時高度被無限放大，徹底壓垮了主圖。
* **預期效果**：
  1. 無論透過什麼順序、是否同時開啟多個副圖，各副圖初始載入時的高度應依據「當前外層容器相對於預設高度 (420px) 的比例」動態給定（亦即 `120px * (當前高度/420px)`）。例如在正常 420px 下加入為 120px；若容器被放大了兩倍至 840px，則新副圖應優雅地以 240px 的姿態入場，維持畫面視覺和諧，絕不可遭到不合理的擠壓。
  2. 即使在不同螢幕解析度或拉大容器的極端狀態下，上述動態分配黃金比例的規則也必須嚴格發揮作用。
  3. 當手動關閉某個副圖時（如關掉 VOL），空出來的高度空間應「全數歸還給主圖」，而畫面上殘存的副圖（如 RSI）必須完美維持其關閉前被賦予的正確高度（例如上述的 240px 或是 120px），絕不可因為計算錯誤而異常暴漲。

### 問題三：展開單一副圖時，未吸收他者空間，反而異常縮小或錯誤侵佔主圖[Not Fixed]
* **現象**：當同時開啟 VOL 與 RSI，並點擊展開 RSI 時，RSI 並不如預期般順利吸收掉隱藏的 VOL 空間。相反地，RSI 展開後的大小反而變得比「展開前 RSI + VOL」的總空間還要小，或是錯誤地暴漲去侵佔了上方主圖 (K線圖) 的空間。
* **根本原因**：
  在「展開(expanded)」模式下，`_captureCurrentPaneHeights` 計算 `scaleFactor` 時，不小心以「展開後的總高度」除以「展開前的部分 Pane 高度總和」，造成放大的 `scaleFactor` 使得主圖的 `_baseMainPaneHeight` 被賦予錯誤的值。此外，在 `_updateSubChartPaneHeights` 中的 expanded 邏輯未能確實把主圖「鎖定」回原大小，導致 LightweightCharts 原生的調整覆蓋了設定，使得展開後的副圖大小變得無法預測。
* **預期效果**：當點擊副圖（如 RSI）的展開按鈕時，主圖的高低大小必須完全鎖死維持原狀。RSI 應該往上長，精確且完整地吸收掉其餘被隱藏的副圖（如 VOL）的空間，不多也不少。

### 問題四：切換篩選結果的股票時，主圖與副圖的高度比例隨機崩壞 (新增)[Not fixed,,still have issue]
* **現象**：在維持相同副圖開啟數量的情況下，當從清單中一檔一檔切換股票（例如從 AA 切到 AAA 再切到 AAAA）時，主圖與副圖之間的高度比例會發生突發性的錯亂。副圖可能會突然往上跳（嚴重壓縮主圖）。即使切換回原本的股票，這種破壞性的比例失衡依然持續存在。
* **根本原因**：當使用者點擊新股票觸發重新渲染 (`renderSubCharts` 取得新資料) 時，系統會先呼叫 `_captureCurrentPaneHeights` 捕捉當前高度。但因為此時舊的圖形實例與新股票資料的載入生命週期發生交錯，ResizeObserver 或 LightweightCharts 內部的高度重繪還沒就緒，導致 `capture` 抓到了錯誤的瞬時高度（例如正在被清除或尚未正確排版的 DOM 尺寸）。這些錯誤的畸形數據被當作 `savedHeight` 寫入 state 後，新股票的圖表就套用了這些被污染的比例，產生了隨機崩壞與下方留白。
* **預期效果**：在維持相同副圖開啟狀態下（例如開啟 VOL 和 RSI），無論怎麼在一檔又一檔不同的股票之間切換，主圖與其餘副圖之間的高度總比例關係應該要完美繼承，絕對不可以出現突發的擠壓、亂跳或是容器下方出現無效黑底留白的情況。

### 問題五：收起左側面板後，圖表上方公司全名未展開仍顯示刪節號 (新增)[Fixed]
* **現象**：當公司名稱過長時（如 `Ambev S.A. American Depositary Shares (Each representing 1 Common Share)`），在左側面板未收起的狀態下，文字尾端顯示 `...`（刪節號）是預期內的正常行為。但當使用者點擊左上角漢堡包按鈕把左側選單收起時，明明上方導航列左側已騰出極大的充裕寬度，公司全名依舊呈現被切斷的刪節號狀態，並未釋放並自動填滿可用的空間。
* **根本原因**：渲染上方圖表標題（公司名稱）的區塊元件，在樣式佈局 (可能是 Tailwind 的 flex 或 w-max / min-w 相關設定) 或是計算可用寬度時，被固定死在某個 max-width，或是未能隨著父層版面 (Layout Sidebar Toggle) 的寬度異動而動態 re-flow 其內容寬界限。
* **預期效果**：當透過左上角漢堡按鈕將左側面板收起，進而使上方圖表 Header 獲得更多實體可用寬度時，過長的公司名稱字串應該要自動展開並利用多出來的空間顯示全名，只在真的沒有空間時才退回顯示刪節號。

## 規格變更協議

為了提供更合理的 UX 並減少無謂的極端擠壓場景，我們與使用者達成以下規格調整：
1. **整體容器最小高度 `_minTotalChartHeight` 提升至 420px**：原來為 300px，現改為與預設高度雷同的 420px。進一步往下壓縮圖表在實務上並無幫助，將其鎖定，即代表使用者「只能拉大，不能讓它比預設更小」。
2. **新增副圖採用相對比例**：承上，新增副圖時不再強制賦值 120px，而是採用計算 `120px * (目前容器高度 / 420px)`，以保持各尺寸螢幕下的主副圖完美比例協調。
