# Screening JS 檔案結構分析與拆分建議

經過對 `App/Feature/Screening/` 目錄下所有 JS 檔案的審查，以下是關於檔案長度、職責分析以及後續重構（拆分）的建議。

## ⚠️ 過長檔案清單 (超過 1000 行)

目前有三個主要檔案體積過大，承擔了過多不同維度的職責：

1.  **`chartSettingsModal.js`** (~1,730 行)
    *   **現狀**：負責所有圖表設定（MA、布林、型態、外觀、座標軸）的渲染、事件綁定，以及一個完整的自定義色板選擇器。
2.  **`screening.js`** (~1,160 行)
    *   **現狀**：作為主控制中心，同時處理網路請求 (SSE)、結果渲染、表格排序、以及頁面佈局調整（拖拉、全螢幕）。
3.  **`chartController.js`** (~1,140 行)
    *   **現狀**：封裝了圖表庫、數據加載、多種指標渲染邏輯、以及極其複雜的 DOM 懸浮窗 (Tooltip) 計算。

### 2. 功能組件現況 (`function_block/` & `indicators/`)
目前專案已將「市場、時間、指標、型態」等區塊拆分為個別 JS，但這些檔案：
- **樣式分散**：所有 CSS 仍堆疊在 `screening.css`。
- **依賴強耦合**：頻繁直接存取 `window.state` 與 `window.ScreeningPage`。
- **目錄結構平鋪**：所有功能塊擠在同一個目錄，不利於擴建。

### 4. 此架構的關鍵運作模式

1.  **功能切片 (Feature Slices)**：
    *   **高內聚**：將原本散落在各處的指標邏輯、UI 與樣式，全部收攏至 `Indicators/` 目錄。這意義著「功能」就是「目錄」，刪除一個資料夾就能乾淨地拔除整個功能。
    *   **Page 外殼** (`screening.html`)：只定義頁面大排版，內容透過 `include` 引入。
    *   **Fragment 容器** (`screening_fragment.html`)：作為 HTMX 的「調度員」，動態組合相關組件。

2.  **HTML 片段化**：
    *   將過長的 `screening.html` (500+ 行) 按照組件拆分到各自的 `templates/ui.html` 中。
    *   在後端 FastAPI 初始化 Jinja2Templates 時，加入多個搜尋目錄：
        ```python
        templates = Jinja2Templates(directory=["App/Feature/Screening/templates", "App/Feature/Screening/Components"])
        ```

3.  **解決 God Object 問題**：
    *   原本 `screening.js` (1100+ 行) 的邏輯會依功能散佈到如 `MarketSelection.js` 與 `ResultsTable.js` 中。
    *   原本 `screening.css` (500+ 行) 將被拆解，讓樣式可以隨著功能組件移動。
將圖表引擎、數據與擴充 Overlay 分離：
- **`ChartCore.js`**：基礎圖表實例管理、多國語言、ResizeObserver。
- **`ChartDataService.js`**：專門負責向 API 請求 K 線數據並進行格式轉換。
- **`ChartIndicatorLayer.js`**：管理掛載在圖表上的各類指標 Layer。
- **`ChartTooltipOverlay.js`**：將那幾百行複雜的「吸附懸浮窗」計算獨立成插件。

## 📈 目前已模組化的部分
值得注意的是，系統中已有一些不錯的模組化實作（如 `StrategyManager.js`, `IndicatorRegistry.js`），建議未來的重構應遵循這種模式，將**邏輯與畫面 (Logic vs UI)** 分得更開。
