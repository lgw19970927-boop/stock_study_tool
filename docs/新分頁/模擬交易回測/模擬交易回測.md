# 模擬交易回測頁面架構設計 (詳細完整版)

頁面整體分為 **左側（回測參數設定面板）** 與 **右側（回測報告與分析面板）** 兩大區塊。左側面板負責定義後端模擬迴圈的所有規則，右側面板則負責接收後端運算結果並進行視覺化渲染，風格排版類似股票篩選頁面的左右側架構。

## ▍左側面板：回測參數設定 (Settings Panel)

此區塊為模擬交易規則的控制中心，所有條件皆在後端進行模擬，獨立運作。

### 1. 基本回測參數 (Basic Parameters)
* **回測期間 (Backtest Period)：**
  * 快捷按鈕列：`1D` / `1W` / `1M` / `3M` / `6M` / `1Y` / `自訂`（風格與股票篩選頁面的「分析時間範圍」一致）。
  * 選擇「自訂」時，展開開始日期 (`start_date`) 與結束日期 (`end_date`) 選擇器。
* **執行頻率 (Execution Frequency)：** * 下拉選單：1分 / 3分 / 5分 / 15分 / 30分 / 1H / 4H / 日 (1D) / 週 (1W) / 月 (1M) / 季。
  * *說明：決定後端回測引擎每次推進的時間單位。*
* **交易方向 (Trade Direction)：** * 單選按鈕：作多 (Long) / 作空 (Short)。
* **執行商品範圍 (Execution Instrument Range)：**
  * 選項卡或下拉選單，決定回測的母體池（僅限美股）：
    1. **股票篩選策略：** 匯入「股票篩選」分頁中已儲存的「我的策略」條件，作為動態股票池（每個時間節點只交易符合該策略的股票）。
    2. **自訂股票清單：** 手動輸入或匯入特定股票代碼（例如：AAPL, NVDA, MSFT）。
    3. **全市場：** 針對美股全市場進行掃描。

### 2. 進場設定 (Entry Settings)
* **進場觸發條件 (Entry Triggers)：**
  * UI 支援**摘要模式 / 編輯模式**雙模式切換（與現有股票篩選器的指標/型態設定一致）：
    * **摘要模式 (Summary Mode)**：顯示簡明條件字串（例如：`MA-日K: MA20 > MA60`），右側附帶「✏ 編輯」與「🗑 刪除」Icon。
    * **編輯模式 (Edit Mode)**：點擊「✏ 編輯」或「+ 新增進場條件」時展開卡片，提供指標 / 型態選擇、週期、比較運算子及數值等下拉選單。
* **進場限制 (Entry Restrictions)：**
  * **開盤跳空設定**：組合式選項 — `開盤跳空` `[向上 / 向下 ▼]` `達` `[輸入框]` `%` 即 `[交易 / 不交易 ▼]`
  * **特定時間區段**：組合式選項 — `開盤後` `[輸入框]` `[分鐘 / 小時 ▼]` `[交易 / 不交易 ▼]`
  * *說明：兩組限制條件為「AND」關係，皆啟用時須同時滿足。*
* **進場價格機制 (Entry Price execution)：**
  * 單選按鈕：`次期開盤價 (Next Open)` / `當期收盤價 (Current Close)` / `條件觸發價 (Trigger Price)`
  * *「條件觸發價」說明：適用於僅有日線資料的環境下模擬「盤中突破」。若當日高 / 低點穿越觸發條件的目標價位，則直接以該突破價格作為進場成本（無需等待下根 K 線開盤）。*
* **最大同時持有檔數 (Max Open Positions)：** * 數字輸入框（留空或輸入 0 代表無限制）。

### 3. 出場設定 (Exit Settings)
* **條件出場 (Condition-based Exit)：**
  * UI 支援**摘要模式 / 編輯模式**雙模式切換（與進場觸發條件一致）。
  * 允許設定技術指標或型態作為出場觸發（例如：跌破 20MA、RSI 死亡交叉）。
* **固定停利 / 停損 (Stop Profit / Stop Loss)：**
  * 停利 (Take Profit)：核取方塊 `[ ]` 啟用，輸入 `%` 數。
  * 停損 (Stop Loss)：核取方塊 `[ ]` 啟用，輸入 `%` 數。
* **時間停損 (Time Stop)：**
  * 最大持有時間：核取方塊 `[ ]` 啟用，輸入 `N` 期（依據上方設定的「執行頻率」單位計算）。
* **出場價格機制 (Exit Price execution)：**
  * 單選按鈕：`次期開盤價 (Next Open)` / `當期收盤價 (Current Close)` / `條件觸發價 (Trigger Price)`
  * *「條件觸發價」說明：同進場機制，適用日線資料下模擬當日觸及出場條件價位時直接出場（例如跌破特定支撐線即以穿破價出場）。*

### 4. 資金與交易設定 (Capital & Trading Settings)
* **初始資金 (Initial Capital)：** * 數字輸入框（例如：$1,000,000）。
* **資金與風險管理模組 (Risk Management Module) [Optional]：**
  * 核取方塊 `[ ]` 啟用 / 停用。
  * *說明：啟用後，後端將自動套用與前端「資金與風險管理」分頁完全相同的數學邏輯。系統會依據設定的「單筆停損容忍度 (%)」自動計算每次進場的建議股數，並監控帳戶總風險。若單筆進場會導致帳戶總風險超標，則自動濾除該筆交易。*
* **交易費用 (Trading Costs)：** * 數字輸入框與單位下拉選單：輸入 `%` 數或單筆固定金額（貼合美股券商常見的計費特性）。

## ▍核心概念說明：篩選頻率 vs 執行頻率 vs 橋接邏輯

下表說明三個相關但獨立的概念，避免使用者在匯入篩選策略時產生混淆：

| 層級 | 名稱 | 定義位置 | 說明 |
|------|------|----------|------|
| 1 | **篩選頻率 (Screening Frequency)** | 股票篩選頁面（儲存在「我的策略」中） | 定義「這個選股策略多久執行一次市場掃描」（例如：每日、每週）。這是策略本身的一部分，與條件內使用的 K 棒週期無關（例如：可以每週掃描，但條件看日 K 線）。 |
| 2 | **執行頻率 (Execution Frequency)** | 模擬回測頁面 | 定義「回測引擎每次前進的時間單位」，決定系統多久去檢查一次進出場條件是否成立。這是回測的屬性，與選股策略無關。 |
| 3 | **橋接邏輯 (Bridge Logic)** | 匯入策略時的處理 | 當回測頁面匯入某個篩選策略時，該策略的「篩選頻率」預填為目前設定的執行頻率，但使用者可以 override。股票篩選頁面本身是監控當前時間點的工具，策略應為自洽的完整描述，因此篩選頻率應保留在策略中，不應從篩選頁面移除。 |

## ▍右側面板：回測報告與分析 (Backtest Report Panel)

點擊「開始回測」後，右側面板將渲染完整報告。建議採用 **頁籤 (Tabs)** 切換或 **區塊向下滾動** 的方式排版。

### 1. 回測基本資訊表頭 (Report Header)
設計為報告最上方的資訊卡，便於快速確認條件或截圖：
* **回測資料範圍：** `YYYY/MM/DD` - `YYYY/MM/DD` (含執行頻率)。
* **回測執行時間：** 報告產生的時間戳記與後端運算耗時。
* **回測執行商品：** 顯示範圍名稱（例：美股全部 / 自訂清單：科技七巨頭），並註記回測成功與失敗的檔數。
* **交易設定摘要：** 顯示交易方向、交易費用、最大持有檔數限制、進/出場價格基準。

### 2. 整體統計表 (Overall Statistics)
核心績效數據總覽，建議包含正負值顏色標示（綠漲紅跌）：
* **交易頻次：** 總交易次數 | 獲利次數 | 虧損次數
* **核心指標：** 勝率 (Win Rate %) | 盈虧比 (Profit/Loss Ratio)
* **報酬與損益分析：** * 總報酬率 (Total Return %)
  * 複合年化成長率 (CAGR)
  * 平均報酬率-單筆 (Avg. Return per Trade %)
  * 平均獲利% (Avg. Win %) | 平均損失% (Avg. Loss %)
  * 平均獲利金額 (Avg. Win $) | 平均損失金額 (Avg. Loss $)
  * 最大獲利-單筆 (Max Profit per Trade %) | 最大虧損-單筆 (Max Loss per Trade %)
* **風險與極端值：** * 最大回撤 (Max Drawdown)
  * 最大連續獲利次數 | 最大連續虧損次數
  * 最大區間獲利率 (Max Interval Profit %) | 最大區間虧損率 (Max Interval Loss %)

### 3. 淨值曲線圖 (Equity Curve)
* **圖表類型：** 互動式折線圖或面積圖。
* **X 軸：** 時間軸（依照回測期間）。
* **Y 軸：** 總報酬率 (%) 或 帳戶總淨值金額。
* **互動功能：** 滑鼠懸停可顯示特定時間點的淨值，圖表中應包含 0% 水平基準線。

### 4. 商品統計表 (Symbol Statistics)
以「單檔股票」為維度的數據表格 (Data Grid)，可點擊欄位標題進行排序：
* 欄位：`股票代碼` | `交易次數` | `勝率` | `總貢獻損益金額` | `總貢獻損益 %`

### 5. 交易明細清單 (Trade Details List)
詳細紀錄每一筆交易的資料表，支援篩選與排序：
* 欄位：
  * `股票代碼`
  * `進場時間` | `進場價格`
  * `出場時間` | `出場價格`
  * `進場原因` (例如：MACD 黃金交叉)
  * `出場原因` (例如：觸發 8% 停損 / 跌破月線)
  * `持有期數`
  * `單筆損益 %` | `單筆損益金額`

### 6. 交易明細圖表 (Interactive Trade Chart)
**[重要互動]** 點擊上方「交易明細清單」中的任一筆列 (Row) 時，此圖表會自動更新為該筆交易的視覺化覆盤：
* **主圖 (K線圖)：** 顯示該檔股票的 K 線走勢。明確標記「進場點 (Buy)」與「出場點 (Sell)」的圖示，並畫出連接進出點的交易線段。
* **副圖 (指標與量能)：** 顯示相關的技術指標線圖與成交量柱狀圖，並畫出對齊進出場時間點的垂直參考線。

## ▍設計審查問答專區（集中版）

### 🔴 衝突 1：執行頻率 vs 股票篩選策略的動態池更新成本
**問題點：** 「執行商品範圍」選擇「股票篩選策略」時，說明指出「每個時間節點只交易符合該策略的股票」。若執行頻率為 `1min` 或 `5min`，回測期間數年，每個時間步都需要重新掃描整個指標篩選策略，計算量會是 `O(T × N × I)` 完全非線性。

**建議：** 需要明確定義股票池的**更新頻率**，例如：
- 每個「執行頻率」步驟更新一次（高成本）
- 每天更新一次（固定）
- 只在回測開始時建立一次靜態池（最快最簡單）

建議在 UI 增加一個選項：「動態更新（每步）」/ 「每日更新」/ 「靜態池（回測開始時建立一次）」。

**Codex答/點評：**
1. 判斷正確，這是最關鍵的計算瓶頸。
2. 建議預設「每日更新」，並允許進階使用者切換「每步更新」。
3. 後端可加快取層：以 `(date_bucket, strategy_hash)` 作為 cache key，減少重複篩選成本。
4. 報告中應回填「股票池更新模式」，避免不同模式結果被誤比較。

**Gemini回答/點評：** 同意Codex的看法，但「每步更新」即使有快取，在長區間與短週期下仍可能導致極大的運算負載。**建議：** 預設採用「每日更新（每日收盤後重新計算次日股票池）」，並要求若週期小於 1D，應在UI跳出效能警示或限制其不能選「每步更新」。

**Claude Opus答/點評：**

同意這是整個回測引擎最核心的效能瓶頸，且 Codex 與 Gemini 的方向大致正確。但我認為三者（每步更新 / 每日更新 / 靜態池）並不應該是平等的三選一，而應該是**階層式限制**：

1. **「靜態池」應為預設值。** 理由：對 MVP 而言，這是唯一能在合理時間內跑完的模式。使用者若選「股票篩選策略」作為母體池，系統在回測 Day 0 執行一次篩選，得到清單後固定到回測結束。這已經能滿足「用篩選條件選出一批股票來回測」的核心需求。
2. **「每日更新」可作為 P2 進階選項。** 它需要後端針對每個交易日都執行一遍篩選策略，計算量等同於「篩選天數 × 全市場篩選成本」。以本系統目前的篩選已有 SSE streaming 機制來看，回測時在背景跑這個是可行的，但要確保有 cache（以 `(trade_date, strategy_config_hash)` 為 key）。
3. **「每步更新」在 MVP 直接砍掉不做。** 即便有 cache，日內頻率 × 策略篩選的組合爆炸在目前的 MySQL 單機架構下不現實。若未來有需求，應搭配預計算的物化視圖或 Redis cache 才能考慮。

**反駁 Gemini** 提出的「效能警示」：光是跳警告不夠，更好的做法是直接在 UI 層把不合理的組合灰掉（disabled），例如當「執行頻率 < 1D」時，從下拉選單中移除「每步更新」選項。

### 🔴 衝突 2：最大持倉數 (Max Open Positions) 與風險管理模組的優先順序未定義
**問題點：** 若「最大同時持有檔數」設為 10，且「資金與風險管理模組」也啟用，兩個系統都可能各自決定「要不要進場」，但優先順序不明：是「先達到 10 檔就停」還是「先計算風險預算再決定是否達到 10 檔限制」？

**建議：** 明確定義執行順序：
1. 觸發進場條件
2. 飛過風險管理模組計算倉位
3. 若通過風險預算，才檢查是否超過最大持倉數
4. 全部通過才實際進場

**Codex答/點評：**
1. 同意你的順序設計，建議正式定義成單一路徑流程，避免實作時分支漂移。
2. 補充必要規則：當候選單超過持倉上限時，要有排序優先級（例如訊號強度、流動性、風險報酬比）。
3. 建議在報告新增「被拒單原因統計」（風險超標 / 持倉滿額 / 流動性不足）。

**Gemini回答/點評：** 同意單一路徑流程。對於超額持倉競爭，建議未來在介面增加「篩選優先級」設定（例如依成交量高低排列）。若不開放設定，預設可使用「依股票代碼字母順序」確保100%可重現性與無隨機性。

**Claude Opus答/點評：**

原文提議的四步流程（觸發 → 風控計算 → 檢查持倉上限 → 實際進場）方向正確，但我認為**順序應微調**：

1. 觸發進場條件 → 產出「候選清單」
2. **先檢查持倉上限**（若已滿額，直接全部拒單，省去後續計算）
3. 對候選清單進行排序（按排序鍵：下述）
4. 依序通過風險管理模組計算倉位大小
5. 若風險預算通過 → 實際進場；若不通過 → 拒單，繼續處理下一個候選

理由：持倉上限是**硬天花板**（O(1) 判斷），風控計算相對昂貴且涉及帳戶狀態查詢。先做便宜的檢查再做貴的，符合短路求值原則。

**關於候選排序鍵：** 反駁 Gemini 預設用「字母順序」的做法。字母排序雖然 100% deterministic，但毫無金融意義——這等於系統性偏好 A 開頭的股票（如 AAPL 永遠優先於 TSLA）。建議 MVP 預設以「當日成交金額（price × volume）降序」排序，既有金融合理性（流動性最佳者優先），又完全 deterministic。

### 🟡 模糊 1：同一個 bar 的進場與出場觸發衝突
**問題點：** 若進場設定為「當期收盤價」，出場設定也為「當期收盤價」，且兩個條件在同一根 K 棒上同時觸發，會產生「同一根棒開倉又平倉，損益 ≈ 0（扣手續費）」的無效交易。

**建議：** 規定若進場條件與出場條件在同一根 K 棒被同時觸發，**優先執行進場**，出場條件延後至下一個時間步驟重新判斷。

**Codex答/點評：**
1. 同意，必須定義 deterministic 規則。
2. 建議新增引擎參數：`same_bar_policy = entry_first|exit_first|ignore_both`，避免硬編單一政策。
3. 報告需註記同 Bar 衝突筆數，便於策略除錯。

**Gemini回答/點評：** 同意Codex的 `same_bar_policy` 的做法。**補充建議：** 在日K線資料中，無法分辨高低點發生的時序。建議加上Tooltips警告使用者「日線內同時觸發條件觸發價，可能產生失真」。實作上建議預設採用 `entry_first` 保守圈存進場資金，或 `ignore_both`。

**Claude Opus答/點評：**

同意必須定義 deterministic 規則。但 Codex 的 `same_bar_policy` 三選一我認為**過度參數化**。回測系統參數已經夠多了，這種「引擎行為設定」不該暴露給使用者。

**我的建議是硬編為 `exit_first`**，理由：
- 在真實交易中，若你持有一檔股票且同根 K 棒同時觸發進場（某策略）與出場（該持倉），合理的操作是**先處理風險（平倉）再考慮新倉**。

- 將此規則寫死在引擎中，並在報告的 metadata 欄位記錄 `same_bar_policy: exit_first`，即可追溯。

同意 Gemini 提出的 tooltip 警告——只要使用者選了「條件觸發價」且為日線，就在 UI 加一行小字提醒日線內無法區分時序。

**MVP 做法：** 在報告表頭固定標記「價格口徑：前復權（Split & Dividend Adjusted）」。若未來有 raw data 需求再開 toggle。

**問題點：** 期末結算口徑若不一致，勝率、回報、交易次數都可能失真。

**建議：** 定義 Force Liquidate 或 Mark-to-Market 模式。

**Codex答/點評：**
1. 部分同意，優先級 P1。

**Gemini回答/點評：** 反駁Codex預設 Force Liquidate（強平）。對於策略而言，強平會在期末吃上一筆滑點與手續費，造成績效扭曲。建議採用 Mark-to-Market（依期末收盤市價預估未實現損益併入淨值）並加上標示，不要創造虛擬的「平倉交易手續費」。

**Claude Opus答/點評：**

這是個比看起來更複雜的問題。我的選擇和 Codex、Gemini 都不完全一樣。

### 🟣 問題 17：Look-ahead bias 與指標暖機期是否已被正確處理
**問題點：** 若指標計算偷看到未來資料，或暖機不足就交易，回測會失真。

**建議：** 明確定義暖機期間、資料截斷規則與可交易起始點。

**Codex答/點評：**
1. 同意，需在引擎層硬性保證。
2. 建議回報每檔標的有效起算時間，避免誤讀策略覆蓋範圍。

**Gemini回答/點評：** 完全同意。技術實作上：假如使用者設定回測開始為 2023/01/01 且設定 MA200，後端引擎必須聰明地往前撈約一年的歷史資料算指標「暖機」。前端 UI 應說明「首筆實際觸發交易將視指標計算的資料起算點而定」。

**Claude Opus答/點評：**

這是回測引擎可信度的**絕對底線**，必須在架構層硬性保證。

看了現有篩選模組的 `screen_single_stock()` 函式，它已經有暖機期計算邏輯（根據指標類型、MA 長度等推算 `total_required_bars`），回測引擎可以直接復用這段邏輯。

**具體建議：**
1. 使用者設定的「回測開始日」是**第一個可交易日**，不是第一個資料日。引擎自動往前延伸所需的暖機期天數來撈取資料，但暖機期間不執行任何進出場判斷。
2. 暖機期長度 = `max(所有進場指標所需週期, 所有出場指標所需週期) + 安全緩衝`。例如用了 MA200，就自動往前撈至少 250 個交易日。
3. **嚴禁在指標計算中使用 `fillna(method='bfill')` 或任何回填未來資料的操作。** 這是 look-ahead bias 最常見的來源。
4. 報告表頭記錄「暖機期：N 個交易日（實際首筆可交易日：YYYY/MM/DD）」。

### 🟣 問題 19：資料缺口、停牌、異常值是否有統一政策

3. 遍歷排序後的清單，依序做風控檢查 + 持倉上限檢查。
4. 通過的進場，不通過的跳過。
5. 若兩個標的成交金額完全相同（極罕見），以 symbol 字母升序作為 tie-break（Gemini 的建議在此作為最後手段是合理的）。

**同意 Codex 在報告中加入「候選/錄取/拒單」統計。** 這不複雜——引擎在處理過程中已經知道候選集與最終錄取集，差值就是拒單數。簡單加兩個計數器即可。

### 🟣 問題 27：回測引擎的執行架構——同步阻塞 vs 非同步背景任務
**問題點：** 回測可能耗時數秒到數分鐘甚至更久。若以 FastAPI 的同步路由處理，會阻塞 worker 線程，影響其他使用者；若以 async 路由 + SSE streaming，長時間運算仍可能觸發 proxy timeout。

**建議：** 採用「提交任務 → 取得 task_id → SSE 訂閱進度」的兩段式設計，將實際運算移至背景線程，確保主 event loop 不被阻塞。

**Codex答/點評：**
1. 兩段式任務流程是必須做的，不建議同步阻塞路由。
2. MVP 不上 Celery，先用 `ThreadPoolExecutor` + 任務登錄表（記憶體）落地，避免引入新基礎設施。
3. API 最少要有四支：`POST /run`（回傳 task_id）、`GET /stream/{task_id}`（SSE）、`GET /status/{task_id}`（輪詢降級）、`POST /cancel/{task_id}`（中止）。
4. `task_id` 用 UUID4；任務狀態至少包含 `queued/running/completed/failed/cancelled`、進度百分比、當前階段、錯誤訊息。
5. Nginx 必加 `proxy_buffering off` 與較長 `proxy_read_timeout`，否則 SSE 會看起來像卡住。

**Gemini回答/點評：** **完全同意採用非同步設計**。在目前的架構下，直接使用背景任務搭配 SSE 確實是性價比最高且最合適的 MVP 方案。不必引入 RabbitMQ 或 Celery 這類重量級依賴。前端可透過 SSE 收到即時的「運算中」狀態推播，若超時或連線中斷也可以透過定時輪詢 `/status/{task_id}` 確保容錯。此外，取消回測的機制 (`cancel`) 強烈建議實作，因為長時間運算若卡死會浪費伺服器大量資源。

**Claude Opus答/點評：**
1. 建議回測 API 分為兩段：`POST /api/backtesting/run` 提交任務（立即回傳 `task_id`），`GET /api/backtesting/stream/{task_id}` 以 SSE 訂閱進度。
2. 實際運算在 `asyncio.to_thread()` 或 `concurrent.futures.ThreadPoolExecutor` 中執行，避免阻塞 event loop。
3. 在 Nginx reverse proxy 層面，SSE 路由需設定 `proxy_buffering off` 與足夠的 `proxy_read_timeout`（如 600s）。本系統的 `env/nginx/` 配置應已有類似設定（篩選 SSE 應該已需要），可複用。
4. 這個架構也天然支援「取消執行」——後端以 `task_id` 查找對應的運算線程 flag，設置取消信號。

**問題點：** 若回測結果只在前端記憶體中，重新整理即遺失，使用者無法回顧歷史回測。但若每次都存庫，回測明細（數千筆交易）會快速膨脹資料量。

**Codex答/點評：**

2. MVP 應持久化「摘要層」：回測參數、核心指標、報告產生時間、param_hash，便於比較與回溯。

**Gemini回答/點評：** **建議 MVP 階段「全不存」以確保開發速度與系統穩定**。回測最注重的是當下的參數驗證，在沒有提供社群分享的 MVP 初期，入庫只會增加後端與資料庫的負擔。針對不想遺失紀錄的使用者，最佳解法是直接提供「匯出成完整報告檔案（JSON 或 CSV）」的功能，讓使用者自行下載儲存。待系統完善、真正確認有歷史比較需求時，再來設計專屬資料表。

1. MVP 建議**不持久化回測結果**。回測是探索性操作，使用者會頻繁調參重跑，大部分中間結果不值得保存。
2. 提供「匯出報告」按鈕（JSON 或 CSV），讓使用者自行保存有價值的結果。
3. P2 考慮在 `user_data` schema 新增 `backtest_reports` 表，但只儲存摘要統計（不含交易明細），且設定自動過期機制（如保留最近 50 份）。

### 🟣 問題 30：「條件觸發價」在日線回測中的數學定義

2. 不建議開放使用者選樂觀/保守模式，避免把日線時序不確定性丟給使用者背責任。

4. 報告必須揭露 `ambiguous_trigger_count`，讓使用者看到這個假設影響範圍。
5. UI 保留「條件觸發價」選項，但在日線模式顯示高亮警告 tooltip。

**Gemini回答/點評：** 在日線級別處理「條件觸發價」注定會有失真問題。我的建議是**「保守容錯與揭露」原則**。在沒有分時資料輔助下遇到同日雙向穿越這類曖昧情況時，若是進場，則直接**略過不進場**；若是出場，則**以保守停損價出場**。此外，必須在回測報告中單獨新增「日線內模糊觸發筆數」的數據展示，讓使用者明確知道有多少次訊號因時序不確定性被略過，以確保策略回報的真實可信度。

**Claude Opus答/點評：**
1. 日線資料中只有 OHLC 四個價格，**完全無法得知盤中的時序**（先到高點還是先到低點）。
2. **MVP 建議的保守做法：** 若 High ≥ 目標價 且 Low ≤ 目標價（同時穿越），則視為「觸發但不確定方向」，此筆信號標記為 `ambiguous`，依設定的 `same_bar_policy` 處理（建議跳過不交易，即 ignore）。
3. 若 High ≥ 目標價 且 Low > 目標價（Low 未下穿），則明確觸發向上突破。
4. 在 UI 中對「條件觸發價」加上 tooltip 警告：「注意：日線資料僅包含 OHLC，無法精確判斷盤中時序。建議搭配其他條件使用，或使用次期開盤價以獲得更確定的結果。」
5. 這個問題是日線回測的根本性限制，不要試圖用啟發式規則（如假設先到 Open 方向的高/低點）來「猜測」盤中走勢——任何猜測都引入了隱藏偏差。

### 🟣 問題 31：回測報告的效能與前端渲染壓力
**問題點：** 若回測產生 5000+ 筆交易明細、500+ 檔股票的商品統計，一次性渲染到 DOM 可能導致前端卡頓或崩潰，特別是本系統採用 Vanilla JS 架構（無 Virtual DOM 框架）。

**建議：** 交易明細採分頁或虛擬滾動；後端 SSE 分段推送大型 payload；淨值曲線圖做降採樣。

**Codex答/點評：**
1. 先解決 80% 問題，不要急著上虛擬滾動套件。
2. MVP 前端策略：交易明細固定分頁（每頁 50 或 100）、商品統計直接渲染、切頁時才更新 table body。
3. 後端策略：SSE 分段回傳（摘要 → 淨值曲線 → 商品統計 → 交易明細 chunks），避免一次吐大 JSON。
4. 淨值曲線超過閾值（例如 3000 點）時由後端做 LTTB 降採樣，保留轉折點。
5. 若後續實測仍卡頓，再評估引入虛擬滾動，而不是先加依賴再找場景。

**Gemini回答/點評：** 在 Vanilla JS 下操作大量 DOM 非常昂貴，**最簡單有效的做法是「傳統分頁」加上「圖表降採樣」**。交易明細列表預設只渲染前 50 或 100 筆，透過按鈕載入下一頁；針對互動式淨值曲線，資料點過多時務必在後端先過濾微小波動，避免圖表縮放明顯掉幀。MVP 階段千萬不要為了解決這個問題去引入虛擬滾動函式庫，這會破壞現有架構的輕量化原則。

**Claude Opus答/點評：**
1. 交易明細表必須實作**虛擬滾動 (Virtual Scrolling)** 或分頁。以本系統現有的 Vanilla JS 架構，建議用簡單的分頁（每頁 50 筆 + 上下頁按鈕），不需要引入虛擬滾動函式庫。
2. 商品統計表通常不超過幾百行，可直接渲染。
3. 淨值曲線圖若資料點超過 2000 個，LightweightCharts 處理得來，但建議後端在回傳時做降採樣（例如每 N 個 bar 取一個資料點），前端 hover 時再載入精確數據。
4. 回測結果的 JSON payload 可能很大（幾 MB），建議後端使用 SSE 分段推送（先推統計摘要讓前端立即渲染，再推交易明細），而非一次性回傳巨大 JSON。

### ⏳ 回測執行中與空狀態 (Progress & Empty State) 生動 UI 設計
為了徹底解決**問題 7** 提到的「等待時間長、卡住感」問題，針對純前端的 MVP 架構（缺少複雜 SSE 推播的狀況下），我們需要在介面上加入以下 UX 設計，確保使用者時刻擁有系統正在運作的安心感：

1. **初始空狀態 (Empty State)：**
   - 右側主畫面一開始不要顯示 `NaN` 或空白圖表。
   - 在正中央居中放置一個脈衝 (Pulse) 動畫大圖示（例如使用主色 `--color-primary` 發光），下方搭配提示文字：「👈 請由左側設定策略參數，然後點擊 開始回測」。

2. **運算中狀態 (Loading / Progress Bar)：**
   - 當按下 `[ ▶ 開始回測 ]` 且 API Request 出去後：
     - 左下角按鈕必須**強制轉變為** `[ ⏹ 取消執行 ]` (邊框顯示閃爍的危險色 `--color-down`)，並阻斷連續點擊。
     - 右側顯示區彈出一個精緻的半透明毛玻璃背景 (Backdrop blur)，中央會浮現出一個**具有真實量化數據的動態進度條 (Progress Bar)**。
     - **真實進度條設計 (需後端 SSE 支援)**：透過後端串流推播，進度條內部會即時顯示填滿的百分比區塊與數值（例如：`68%`）。
     - **預估時間計算邏輯 (ETA)**：利用前端接收到的每一次 `(進度增量, 耗時增量)` 計算平均速度，並即時運算推導出「預估剩餘時間（約 12 秒）」，顯示於條塊上方，給予最精確的倒數期盼感。
     - **文字安定感設計 (Micro-copy)**：在閃爍進度軌的下方，每 2 秒自動輪播洗腦文字（例如：「正在向資料庫撈取歷史 K 線...」➜「正在逐日掃描進出場條件...」➜「模擬撮合與結算中...」）。就算有真實進度條，這項**純前端魔法**依然可以維持畫面的豐富動態感，能讓使用者的等待體感時間銳減 50%！非常建議保留此設計。

3. **中斷與取消確認狀態 (Cancel Confirmation Modal)：**
   - 點選「⏹ 取消執行」後，畫面中央跳出遮罩對話框（右上角附帶 ✖ 關閉按鈕，點擊行為等同繼續執行）。
   - 對話框內包含提示語與 `[☑] 保留並顯示目前已完成的運算結果` 勾選框，並仿照篩選頁面提供 `[ 取消 ]` 與 `[ 確定 ]` 雙按鈕。
   - 若使用者勾選並按下「確定」，前端會傳送中斷訊號給後端（類似篩選頁面的停止機制），後端隨即提前結算目前已完成的進度，並將**提早結算的部分回測報表**回傳給前端。
   - 前端在渲染這份不完整報告時，必須在報表表頭加入醒目的 `[⚠ 本報告為提前中斷之部分結果]` 警告標示，避免未來引發策略效度誤讀。若未勾選並按下確定，則徹底放棄目前計算結果，返回「初始空狀態」。

7. **「開始回測」按鈕**：固定在左側面板底部，使用現有系統的 `--accent` 顏色。必須設定 `position: sticky; bottom: 0; z-index: 10;` 使其永遠貼底懸浮確保隨時可見，避免使用者需要往下捲動才能按；執行中應配合顯示進度條（SSE streaming），且支援中途停止。

> **程式碼結構即商業邏輯結構**：回測頁面的每個業務概念（回測引擎、策略設定、報表渲染）對應一個獨立模組，模組間只透過明確 API 介面溝通。

| 原則 | 回測頁面中的具體意義 |
|------|---------------------|
| **高內聚** | 回測引擎邏輯、策略解析、報表計算各自獨立，不混寫在同一服務層函數中 |
| **低耦合** | 前端回測設定 UI 只呼叫 `/backtest/run` API，不直接依賴篩選模組內部邏輯 |
| **單一責任** | `backtest_engine.py` 只管逐筆模擬；`report_builder.py` 只管績效指標計算 |
| **共用邏輯集中** | 與篩選頁共用的指標計算（MA、BOLL）應透過 `indicators/shared/` 引用，不重複實作 |

### 前端樣式規範

> 統一採用 **Tailwind CSS** 作為樣式框架，禁止在回測新頁面引入自訂 CSS class（除非 Tailwind 無法涵蓋的動態樣式需使用 CSS 變數）。與現有篩選頁保持一致的組件風格（卡片尺寸、色彩變數 `--color-up` / `--color-down`、resizable panel）。

## ▍效能風險警告

### O(T × N × I) 複雜度分析

當回測設定啟用「動態篩選池（每日重新篩選）」時，運算複雜度為：

```
O(T × N × I)
  T = 回測天數（例：252 天/年）
  N = 候選股票數量（例：500 支標普 500）
  I = 每支股票的指標計算量（MA、BOLL 等）
```

若 T=252, N=500, I=5 條指標 → **≈ 630,000 次指標計算**，可能造成嚴重效能瓶頸。

### 警告情境

| 情境 | 複雜度 | 風險等級 |
|------|--------|---------|
| 靜態股票池（固定清單）× 回測 | O(N × T) | 🟢 低 |
| 動態篩選池（每日篩選）× 回測 | O(T × N × I) | 🔴 高 — 接近 O(N²) 等級 |
| 多策略批量回測（M 個策略） | O(M × T × N × I) | 🔴 極高 |

### 建議緩解方案

1. **預設使用靜態股票池**：使用者手動輸入股票代碼或從篩選結果一次複製，而非每日動態重篩
2. **指標結果快取**：若啟用動態篩選池，對每支股票的歷史指標資料進行快取（以日期為 key），避免重複計算
3. **進度條 + 可中止**：回測執行時必須提供 SSE streaming 進度條，並支援「停止回測」按鈕，防止使用者長時間等待無反應界面
4. **批量上限限制**：動態篩選池模式建議限制最多 N=100 支股票 + T=252 天（即 1 年），超出時提示使用者切換為靜態池
