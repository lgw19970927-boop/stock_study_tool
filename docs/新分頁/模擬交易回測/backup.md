## 🔴 衝突 1：執行頻率 vs 股票篩選策略的動態池更新成本

**Claude Opus答/點評：**

同意這是整個回測引擎最核心的效能瓶頸，且 Codex 與 Gemini 的方向大致正確。但我認為三者（每步更新 / 每日更新 / 靜態池）並不應該是平等的三選一，而應該是**階層式限制**：

1. **「靜態池」應為預設值。** 理由：對 MVP 而言，這是唯一能在合理時間內跑完的模式。使用者若選「股票篩選策略」作為母體池，系統在回測 Day 0 執行一次篩選，得到清單後固定到回測結束。這已經能滿足「用篩選條件選出一批股票來回測」的核心需求。
2. **「每日更新」可作為 P2 進階選項。** 它需要後端針對每個交易日都執行一遍篩選策略，計算量等同於「篩選天數 × 全市場篩選成本」。以本系統目前的篩選已有 SSE streaming 機制來看，回測時在背景跑這個是可行的，但要確保有 cache（以 `(trade_date, strategy_config_hash)` 為 key）。
3. **「每步更新」在 MVP 直接砍掉不做。** 即便有 cache，日內頻率 × 策略篩選的組合爆炸在目前的 MySQL 單機架構下不現實。若未來有需求，應搭配預計算的物化視圖或 Redis cache 才能考慮。

**反駁 Gemini** 提出的「效能警示」：光是跳警告不夠，更好的做法是直接在 UI 層把不合理的組合灰掉（disabled），例如當「執行頻率 < 1D」時，從下拉選單中移除「每步更新」選項。

## 🔴 衝突 2：最大持倉數 (Max Open Positions) 與風險管理模組的優先順序未定義

**Claude Opus答/點評：**

原文提議的四步流程（觸發 → 風控計算 → 檢查持倉上限 → 實際進場）方向正確，但我認為**順序應微調**：

1. 觸發進場條件 → 產出「候選清單」
2. **先檢查持倉上限**（若已滿額，直接全部拒單，省去後續計算）
3. 對候選清單進行排序（按排序鍵：下述）
4. 依序通過風險管理模組計算倉位大小
5. 若風險預算通過 → 實際進場；若不通過 → 拒單，繼續處理下一個候選

理由：持倉上限是**硬天花板**（O(1) 判斷），風控計算相對昂貴且涉及帳戶狀態查詢。先做便宜的檢查再做貴的，符合短路求值原則。

**關於候選排序鍵：** 反駁 Gemini 預設用「字母順序」的做法。字母排序雖然 100% deterministic，但毫無金融意義——這等於系統性偏好 A 開頭的股票（如 AAPL 永遠優先於 TSLA）。建議 MVP 預設以「當日成交金額（price × volume）降序」排序，既有金融合理性（流動性最佳者優先），又完全 deterministic。

## 🟡 模糊 1：同一個 bar 的進場與出場觸發衝突

**Claude Opus答/點評：**

同意必須定義 deterministic 規則。但 Codex 的 `same_bar_policy` 三選一我認為**過度參數化**。回測系統參數已經夠多了，這種「引擎行為設定」不該暴露給使用者。

**我的建議是硬編為 `exit_first`**，理由：
- 在真實交易中，若你持有一檔股票且同根 K 棒同時觸發進場（某策略）與出場（該持倉），合理的操作是**先處理風險（平倉）再考慮新倉**。
- 對於「尚未持有」的新標的，不存在出場條件觸發的問題，所以只有「已持有且進場條件又觸發」這個邊界才需要處理。
- 將此規則寫死在引擎中，並在報告的 metadata 欄位記錄 `same_bar_policy: exit_first`，即可追溯。

同意 Gemini 提出的 tooltip 警告——只要使用者選了「條件觸發價」且為日線，就在 UI 加一行小字提醒日線內無法區分時序。

## 🟡 模糊 2：時間停損的顯示單位不夠直觀

**Claude Opus答/點評：**

完全同意，這是低成本高效益的 UX 改善。但我認為 Codex 建議的「同時顯示交易日與自然日」會讓介面過於囉嗦。

**建議只顯示自然日估算即可**，例如：`12 期 ≈ 約 84 天（12 週）`。理由：
- 使用者設定「最大持有時間」的心智模型是「我最多願意持有多久」，這是自然日的概念。
- 交易日換算對一般使用者沒有額外價值，頂多在 tooltip 中補充。

實作上只需一個純前端的 `input` 事件監聽器，根據目前的執行頻率做乘法換算，零後端成本。

## 🟡 模糊 3：回測期間資料可用性驗證缺失

**Claude Opus答/點評：**

同意需要 preflight validation。Gemini 提出的「在 Datepicker 上直接 disable 無資料日期」方向很好，但實作成本較高——需要後端提供一個 API 回傳「各 timeframe 的最早/最晚可用日期」，且 Datepicker 需要自訂 disable 邏輯。

**我的建議分兩階段：**
1. **MVP（低成本）：** 按下「開始回測」時，後端先做一次快速查詢 `SELECT MIN(datetime), MAX(datetime) FROM market_data_ohlcv WHERE timeframe = ?`，若使用者設定的區間完全在可用範圍外 → 直接阻止並報錯；若部分超出 → 自動裁切至可用範圍，在報告表頭醒目標示「⚠ 實際回測區間已自動調整為 YYYY/MM/DD - YYYY/MM/DD」。
2. **P2 改善：** 頁面載入時即非同步取得可用區間，顯示在日期選擇器旁的灰字提示（如 `可用資料：2019-01-02 ~ 2026-03-31`），讓使用者自行判斷。

反駁 Codex 的「硬錯誤 / 軟警告」分級——對 MVP 來說，只需要兩種：阻止（完全無資料）與自動修正（部分有資料），不需要更複雜的分級。

## 🟡 模糊 4：CAGR 的計算邊界情況

**Claude Opus答/點評：**

同意 CAGR 在短期回測下會嚴重誤導。Gemini 的「不足一年直接顯示 N/A」我認為**過於極端**——3 個月的回測還是可以年化，只要使用者理解其限制。

**我的建議：**
- **主指標永遠是 Total Return %**，不論回測區間長短。
- CAGR 永遠計算並顯示，但 **回測區間 < 1 年時，數值旁邊自動附加 `*` 號**，圖表卡片底部顯示小字註解：「* 回測期間不足一年，年化數值僅供參考」。
- 額外增加 **Max Drawdown** 作為第二主指標。單看回報率而不看回撤是典型的新手陷阱，把 Max Drawdown 提升到與 Total Return 並列能建立正確的風險意識。

反駁 Codex 的「日化報酬分布」——對散戶使用者來說，報酬分布直方圖是進階量化工具，放在 MVP 會增加認知負擔。

## 🟠 問題 7：回測等待時間長，是否需要完整狀態回饋

**Claude Opus答/點評：**

**同意 Gemini 的反駁：MVP 只需 3 種狀態 `idle → running → completed/failed`。**

Codex 的 6 狀態機在 Vanilla JS 架構下確實是過度設計。但我想補充：即使只有 3 狀態，「running」狀態的 UX 仍然極為重要：

1. **後端 SSE 推播真實進度是最高優先級。** 本系統篩選模組已有 `/api/screening/filter/stream` 的 SSE 實作經驗，回測模組可以直接複製該模式。後端在遍歷每個交易日時推送 `{progress: 0.68, current_date: "2024-06-15"}`，前端渲染進度條。
2. **取消機制必做。** 回測可能耗時數分鐘，沒有取消等於綁架使用者。後端用一個 `asyncio.Event` 或全域 flag 即可實現。
3. **輪播提示文字我認為不必要。** 它是純裝飾性的「假進度」，有了真實 SSE 進度後，假文字反而降低專業感。如果沒有 SSE，那輪播文字才有存在意義。

## 🟠 問題 8：交易明細 K 線圖是否要直接做瀏覽器全螢幕

**Claude Opus答/點評：**

**同意 Gemini 的做法：直接用瀏覽器原生 `element.requestFullscreen()` API。**

Codex 擔心的 z-index、overflow 問題在自定義面板最大化中確實是地獄，而且本系統是深色主題，全螢幕後圖表背景色天然契合，不會有突兀感。

實作上只需要：
```js
document.getElementById('trade-chart-container').requestFullscreen();
```
加上一個 `Esc` 退出（瀏覽器原生支援）和一個浮動的「退出全螢幕」按鈕即可。

**但補充一點：** 需要在 `fullscreenchange` 事件中呼叫 LightweightCharts 的 `chart.resize()` 或 `chart.applyOptions()`，否則圖表不會自動填滿全螢幕容器。這是 LightweightCharts 的已知行為，篩選頁面的 resize handle 應該已經處理過類似邏輯，可以複用。

## 🟠 問題 10：交易費用以外，是否需加入滑點與流動性限制

**Claude Opus答/點評：**

同意滑點必做。但 Codex 的三個參數（`slippage_bps`、`max_fill_pct_of_bar_volume`、`min_dollar_volume_filter`）一次全上太重。

**我的優先級建議：**
1. **MVP 必做：滑點 (slippage)**。UI 上只需一個輸入框：`滑點估計：[ ] bps`（預設填 5bps）。後端在每次成交時，做多加滑點、做空減滑點。這個改動對引擎的侵入性最小。
2. **P2：成交量限制 (`max_fill_pct_of_bar_volume`)**。限制單筆進場量不超過當 bar 成交量的 N%。但需注意——若使用者用的是日線，一天只有一根 bar，volume 通常很大，這個限制只對小型股有意義。
3. **P3：部分成交。** 這需要引擎支援「部分倉位」概念，大幅增加複雜度（交易明細要記錄填充率、剩餘委託要追蹤……）。MVP 不做。

## 🟠 問題 11：除權息/分割與價格口徑是否要明確定義

**Claude Opus答/點評：**

**強烈同意 Gemini：系統應該強制統一使用調整後價格 (Adjusted Price)，不給使用者選項。**

理由：
1. 本系統的資料來源多半是 yfinance 或類似 API，它們預設回傳的就是 split-adjusted close。看了資料庫 schema，`market_data_ohlcv` 存的應該已經是調整後價格。
2. 給使用者選「Raw」是純粹挖坑——用 Raw price 算 MA 會在分割日產生巨大斷裂，任何策略結果都不可信。
3. 「Total-return adjusted」（含股息再投入）需要額外的股息資料表，目前 schema 中沒有看到，先不支援。

**MVP 做法：** 在報告表頭固定標記「價格口徑：前復權（Split & Dividend Adjusted）」。若未來有 raw data 需求再開 toggle。

## 🟠 問題 12：資金單利/複利邏輯是否必須顯式選擇

**Claude Opus答/點評：**

同意必須顯式選擇。Gemini 的用語轉換很好（「固定金額」vs「淨值 N%」），但我想進一步簡化 UI：

**建議設計為「部位大小模式」(Position Sizing Mode)，單選按鈕：**
- **固定金額 (Fixed Amount)：** 每次進場投入固定美元數（例如 $10,000 / 筆）。→ 這是「恆量單利」。
- **帳戶比例 (% of Equity)：** 每次進場投入當前帳戶淨值的 N%。→ 這是「複利」。
- **風控模組自動計算：** 若上方啟用了風險管理模組，則部位大小完全交由風控公式決定。此選項灰掉前兩者。

這樣三個選項互斥、語意清晰，且與風控模組的關係一目了然。

## 🟠 問題 13：回測期末未平倉部位該如何結算

**Claude Opus答/點評：**

這是個比看起來更複雜的問題。我的選擇和 Codex、Gemini 都不完全一樣。

**我的建議是：MVP 固定採用 Mark-to-Market，但在統計指標中將已平倉與未平倉分拆。**

理由：
- Gemini 說得對，Force Liquidate 會產生一筆不存在的手續費與滑點成本，這對策略評估不公平。
- 但 Codex 也有道理，Mark-to-Market 的未實現損益若直接併入勝率計算，會混淆「已完成交易」的統計。

**具體做法：**
- 回測結束時，仍持有的部位以最後一根 bar 的收盤價計算未實現損益。
- 整體統計中的「勝率」、「盈虧比」、「平均獲利/損失」只計算**已平倉交易**。
- 「總報酬率」、「淨值曲線」、「最大回撤」則包含未實現損益。
- 在報告表頭標記：「期末未平倉 N 筆，以 Mark-to-Market 計入淨值，未計入勝率統計」。
- 交易明細中，未平倉部位的「出場時間」欄位顯示 `持倉中` 標籤，「出場原因」顯示 `回測期末（未平倉）`。

## 🟠 問題 14：做空是否要同時引入借券費與保證金機制

**Claude Opus答/點評：**

同意先做 Cash Account、暫不做 Margin。

**關於借券費：我認為 MVP 也不必做。** 理由：
1. 借券費率（borrow rate）隨標的、隨時間浮動極大（某些 Hard-to-Borrow 股票 borrow rate 可達年化 50%+），使用者自行輸入一個「年化約當費率」只是自欺欺人，算出來的結果仍然不貼近現實。
2. 真正關鍵的成本在主要大型股上通常只有 0.25%～1%/年，對回測結果的影響遠小於滑點。
3. MVP 先讓做空功能能跑、報表能看，在「交易費用」欄位中讓使用者自行把預估的借券成本加進百分比手續費即可（例如手續費設 0.1% + borrow 成本 0.3% = 設 0.4%）。

**待 P2 有真實 borrow rate 資料來源時，再做獨立欄位。**

## 🟠 問題 15：參數暫存與策略模板是否屬必做項

**Claude Opus答/點評：**

**部分同意 Gemini 反駁 LocalStorage。但 Gemini 的「每次參數變動都非同步存資料庫」也過於激進。**

本系統已有 `user_data.strategies` 表（`configuration` 為 LONGTEXT JSON），策略保存的後端基建已存在。

**我的建議是兩層設計：**
1. **草稿暫存用 LocalStorage（自動、靜默）：** 使用者修改參數時 debounce 500ms 自動存入 `localStorage`，重新整理或意外關閉後可恢復。這層是免費的保險，不需要後端。一般使用者 99% 的操作只在一台裝置上，清快取遺失草稿是極小概率事件。
2. **策略模板用資料庫（顯式、使用者主動觸發）：** 點擊「儲存為模板」按鈕時，以 POST API 存入 `strategies` 表（可能需要加一個 `type` 欄位區分篩選策略/回測策略）。這與篩選頁面的「我的策略」體驗一致。

不需要「每次參數變動都寫資料庫」——以本系統的參數量（幾十個欄位），每秒可能觸發數次寫入，既浪費資料庫連線又增加延遲。

**Schema version 必做**——`{schema_version: 1, ...params}` 包在 JSON 最外層即可。

## 🟣 問題 16：生存者偏差（Survivorship Bias）是否會扭曲結果

**Claude Opus答/點評：**

同意 Gemini 的務實做法。看了資料庫 schema，`stock_meta` 表有 `status` 欄位（`Active`, `Delisted`, `Suspended`），但 `market_data_ohlcv` 中退市股的歷史資料是否完整保留，取決於資料同步服務的邏輯。

**我的建議：**
1. MVP 在回測報告表頭加一行固定免責聲明：「⚠ 本回測基於目前可取得之歷史市場資料，不保證涵蓋已退市標的，長週期回測可能存在生存者偏差。」
2. 後端在產生股票池時，應查詢 `stock_meta` 中 `status = 'Delisted'` 的標的，若其在回測區間內有 OHLCV 資料，**應納入回測**（而非只篩 Active 標的）。
3. 報告表頭增加一行：「股票池含退市標的 X 檔 / 不含 Y 檔」。

## 🟣 問題 17：Look-ahead bias 與指標暖機期是否已被正確處理

**Claude Opus答/點評：**

這是回測引擎可信度的**絕對底線**，必須在架構層硬性保證。

看了現有篩選模組的 `screen_single_stock()` 函式，它已經有暖機期計算邏輯（根據指標類型、MA 長度等推算 `total_required_bars`），回測引擎可以直接復用這段邏輯。

**具體建議：**
1. 使用者設定的「回測開始日」是**第一個可交易日**，不是第一個資料日。引擎自動往前延伸所需的暖機期天數來撈取資料，但暖機期間不執行任何進出場判斷。
2. 暖機期長度 = `max(所有進場指標所需週期, 所有出場指標所需週期) + 安全緩衝`。例如用了 MA200，就自動往前撈至少 250 個交易日。
3. **嚴禁在指標計算中使用 `fillna(method='bfill')` 或任何回填未來資料的操作。** 這是 look-ahead bias 最常見的來源。
4. 報告表頭記錄「暖機期：N 個交易日（實際首筆可交易日：YYYY/MM/DD）」。

## 🟣 問題 18：時區、交易日曆、半日市處理是否一致

**Claude Opus答/點評：**

同意 Gemini 統一用 US/Eastern 的做法。

**但我要補充一個本系統的現實約束：** 目前資料庫中 `market_data_ohlcv.datetime` 是 `DATETIME` 型別而非 `TIMESTAMP`，不帶時區資訊。看資料同步模組（yfinance），日線資料的 datetime 通常是 `YYYY-MM-DD 00:00:00`，本身就沒有盤中時間概念。

**MVP 做法：**
1. 日線回測完全不涉及盤中時間，不需要處理 DST、半日市。一根 bar = 一個交易日，以日期為 key 即可。
2. 交易日曆可用 `pandas.tseries.offsets.CustomBusinessDay` 搭配 `pandas_market_calendars` 或簡單的硬編碼美股假日清單來驅動。MVP 用硬編碼假日清單就夠了。
3. 若未來支援分時資料，再引入 `exchange_calendars` 套件做精確的交易時間窗管理。

## 🟣 問題 19：資料缺口、停牌、異常值是否有統一政策

**Claude Opus答/點評：**

同意 Gemini 的 Forward Fill 方案。但必須精確定義 Forward Fill 的行為：

1. **停牌日：** OHLC 沿用前一個交易日的收盤價（O=H=L=C=前收），Volume = 0。這確保指標計算不會斷裂，但 Volume = 0 會自然觸發成交量限制（若啟用），避免在停牌日產生交易。
2. **資料缺口（非停牌，純粹是資料遺失）：** 現有 schema 已有 `data_gaps` 表追蹤。回測引擎遇到缺口時，同樣 Forward Fill，但應在報告的資料品質摘要中記錄：「⚠ XX 檔股票存在資料缺口，共 Y 個交易日使用前值填充」。
3. **異常值：** MVP 不做過濾。極端跳價（如 2020-03-16 美股熔斷）是真實市場事件，不應被剔除。若使用者想排除特定事件，可以調整回測區間。

**不需要在引擎中建立複雜的異常值偵測管線——這是資料品質問題，應在 data_sync 層面解決，而非回測層面。**

## 🟣 問題 20：成交模型（市價/限價/部分成交）是否可重現

**Claude Opus答/點評：**

同意必須 100% deterministic。**但 MVP 根本不需要複雜的成交模型。**

**MVP 的成交模型極其簡單：**
1. 所有委託都是「市價單」，即只要觸發條件，下一步就成交，不存在撮合問題。
2. 成交價格由使用者選的「進/出場價格機制」決定（次期開盤、當期收盤、條件觸發價），三者皆為確定值。
3. 不支援限價單、不支援部分成交。
4. 滑點以固定 bps 加減在成交價上。

在這個簡化模型下，同參數同資料永遠出同結果（deterministic），不需要 tie-break 規則或隨機種子。

**完全同意 Gemini 反對引入隨機種子。** 回測結果不可重現 = 回測系統無意義。

## 🟣 問題 21：多標的同時觸發時，資金競爭規則是否固定

**Claude Opus答/點評：**

此題與衝突 2 部分重疊。我在衝突 2 已建議以「成交金額降序」作為候選排序鍵。

**補充說明排序流程：**
1. 某個時間步中，收集所有觸發進場條件的候選標的。
2. 按成交金額（前一交易日的 close × volume）降序排列。
3. 遍歷排序後的清單，依序做風控檢查 + 持倉上限檢查。
4. 通過的進場，不通過的跳過。
5. 若兩個標的成交金額完全相同（極罕見），以 symbol 字母升序作為 tie-break（Gemini 的建議在此作為最後手段是合理的）。

**同意 Codex 在報告中加入「候選/錄取/拒單」統計。** 這不複雜——引擎在處理過程中已經知道候選集與最終錄取集，差值就是拒單數。簡單加兩個計數器即可。

## 🟣 問題 22：可重現性（Reproducibility）是否可追溯到版本層級

**Claude Opus答/點評：**

同意報告應附 metadata。但 Codex 與 Gemini 提出的欄位有些過度：

**MVP 必做的 metadata：**
- `param_hash`：所有回測參數的 SHA-256 雜湊，用於快速比較兩份報告是否用了相同參數。
- `engine_version`：硬編碼在後端程式碼中的版本字串（如 `"1.0.0"`）。
- `executed_at`：報告產生的 ISO 8601 時間戳。
- `actual_date_range`：實際使用的資料日期範圍（可能經 preflight 調整）。

**不需要的：**

**關於匯出 JSON：** 同意 Gemini 這是好方向，但 MVP 先做「匯出報告為 JSON」就好，「載入別人的策略」是社群功能，優先級極低。

## 🟣 問題 23：過度擬合治理是否納入流程

**Claude Opus答/點評：**

**強烈同意 Gemini 的反駁：MVP 不做 OOS 或 walk-forward。**

理由：
1. Walk-forward 需要多次回測迭代，計算成本是單次回測的 N 倍，目前架構吃不消。
2. OOS 設定（如拆 70/30）看似簡單，但「該用多少比例拆」、「拆幾次」本身就是一個需要專業知識才能回答的問題，反而增加使用者混亂。

**MVP 唯一要做的事：** 在文件或 tooltip 中提醒使用者「建議將回測區間切分為訓練期與驗證期，分別觀察績效穩定性，以避免過度擬合。」

## 🟣 問題 25：參數版本相容與前後端 schema 演進

**Claude Opus答/點評：**

同意 Gemini 的做法。具體實作建議：

1. 所有回測參數的 JSON payload 最外層包含 `schema_version` 欄位（整數，從 `1` 開始）。
2. 後端接收到回測請求時，先檢查 `schema_version`：
   - 等於當前版本 → 直接處理。
   - 低於當前版本 → 嘗試 migration（用一個 `migrate_v1_to_v2()` 的函式鏈），成功後繼續。

4. 從資料庫讀取舊模板時，前端在載入後顯示 toast：「此模板建立於舊版，部分設定已調整為預設值」。

這種 migration 機制在 Django、Rails 等框架中是標準做法，實作成本不高。

### 🟣 問題 27：回測引擎的執行架構——同步阻塞 vs 非同步背景任務

**問題點：** 回測可能耗時數秒到數分鐘甚至更久。若以 FastAPI 的同步路由處理，會阻塞 worker 線程，影響其他使用者。若以 async 路由 + SSE streaming，長時間運算仍可能觸發 proxy timeout。

**Claude Opus答/點評：**
1. 建議回測 API 分為兩段：`POST /api/backtesting/run` 提交任務（立即回傳 `task_id`），`GET /api/backtesting/stream/{task_id}` 以 SSE 訂閱進度。
2. 實際運算在 `asyncio.to_thread()` 或 `concurrent.futures.ThreadPoolExecutor` 中執行，避免阻塞 event loop。
3. 在 Nginx reverse proxy 層面，SSE 路由需設定 `proxy_buffering off` 與足夠的 `proxy_read_timeout`（如 600s）。本系統的 `env/nginx/` 配置應已有類似設定（篩選 SSE 應該已需要），可複用。
4. 這個架構也天然支援「取消執行」——後端以 `task_id` 查找對應的運算線程 flag，設置取消信號。

### 🟣 問題 30：「條件觸發價」在日線回測中的數學定義

**問題點：** 設計文件提到「條件觸發價」適用於日線資料模擬盤中突破，但未精確定義如何確定突破發生的先後順序。例如：使用者設定「突破 $150 進場」，今日 K 棒 High=$155、Low=$140。既觸碰了 $150 以上（可能突破），又觸碰了 $150 以下（可能假突破回落），引擎該如何判定？

5. 這個問題是日線回測的根本性限制，不要試圖用啟發式規則（如假設先到 Open 方向的高/低點）來「猜測」盤中走勢——任何猜測都引入了隱藏偏差。

### 🟣 問題 31：回測報告的效能與前端渲染壓力

**問題點：** 若回測產生 5000+ 筆交易明細、500+ 檔股票的商品統計，一次性渲染到 DOM 可能導致前端卡頓或崩潰。

**Claude Opus答/點評：**
1. 交易明細表必須實作**虛擬滾動 (Virtual Scrolling)** 或分頁。以本系統現有的 Vanilla JS 架構，建議用簡單的分頁（每頁 50 筆 + 上下頁按鈕），不需要引入虛擬滾動函式庫。
