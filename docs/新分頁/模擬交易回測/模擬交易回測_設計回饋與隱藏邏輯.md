# 模擬交易回測頁面 - 設計回饋與潛在隱藏邏輯

本文件為針對《模擬交易回測頁面架構設計 (詳細完整版)》的 UI/UX 排版優化建議，以及實作上容易被遺漏的底層邏輯與隱藏功能梳理。

---

## 一、 UI/UX 排版與設計優化建議（保持既有系統風格）

為了讓畫面看起來專業且不擁擠，建議在現有左右雙欄的基礎上，加入以下細節調整：

### 1. 漸進式揭露 (Progressive Disclosure) 減輕視覺負擔
* **現狀**：左側面板需要設定的參數極多（包含基本、進場、出場、資金管理）。
* **優化**：將進階、不常改動的配置（如：「進場限制的跳空設定」、「時間停損」、「次/當期開盤價機制」）收合在 `[+] 顯示進階設定` 的折疊面板 (Accordion/Details) 內。確保使用者第一眼只看到最核心的參數：期間、頻率、觸發條件、初始資金。

**Codex答/點評：**
1. 這項建議正確，且優先級高（P1）。
2. 建議將左側設定拆成「核心設定（預設展開）」與「進階設定（預設收合）」，避免首次使用者認知超載。
3. 實作時要補上狀態記憶（記住 Accordion 展開狀態），避免每次切頁都重置。

### 2. 運行前與運行中的狀態回饋 (States Feedback)
* **空狀態 (Empty State)**：在使用者尚未點擊「開始回測」前，右側大面積不應只是全黑或空白。建議放上一個符合系統風格的提示卡片或插圖（例如：「請於左側設定條件，並點擊開始」），引導使用者行為。
* **加載進度 (Loading State)**：回測動輒處理十年以上的 K 線數據，耗時較長。建議在「開始回測」按鈕操作後，除了按鈕變成 Spinner 外，右側報表區可顯示**骨架屏 (Skeleton Screen)**，或是顯示文字進度（如：「正在載入美股資料...」、「正在計算交易邏輯...」），以降低使用者的等待焦慮。

**Codex答/點評：**
1. 同意，且應建立明確狀態機：`idle -> validating -> running -> partial -> done -> failed`。
2. 除 Skeleton 外，建議補「可中止回測」按鈕與進度階段文字，降低長任務焦慮。
3. 若後端採 SSE，前端需處理 reconnect 與中斷回復，避免 UI 卡在 loading。

### 3. 右側報表的空間利用與沉浸式圖表
* **現狀**：交易明細 Tab 預設使用上下分割（上列表、下K線），雖有拖曳條，但在筆記型電腦螢幕上依然可能顯得侷促。
* **優化**：建議在下方的 K 線圖卡片右上角，增加一個 **「全螢幕 / 最大化 (Maximize)」** 按鈕。點擊後可以讓該筆交易的 K 線圖覆蓋整個右側面板甚至全螢幕，方便使用者仔細核對技術指標與進出場點的細節。

**Codex答/點評：**
1. 同意，建議優先做「右側面板最大化」再做「瀏覽器全螢幕」，降低實作風險。
2. 建議沿用既有 chart full-screen class 切換模式（與現有 K 線區行為一致），避免新邏輯分叉。
3. 要補鍵盤與可及性：`Esc` 退出、按鈕 `aria-label`、focus 回到來源列。

### 4. 視覺層次與顏色暗示
* **交易方向**：左側的「作多 / 作空」選項，可以考慮在選中「作多」時附帶微弱的綠色或系統主色邊框暗示；「作空」帶紅色邊框暗示，讓使用者一眼就能確認目前的策略方向。

**Codex答/點評：**
1. 同意，但建議不要只靠顏色，需同時保留文字與 icon（避免色弱辨識問題）。
2. 建議使用既有色彩 token（`--color-success` / `--color-danger`）與低飽和背景，不要破壞現有深色系統風格。

## 二、 隱藏邏輯與計劃書未提及的關鍵功能 (Hidden Logic)

在打造真實可信的金融回測系統時，通常還會遇到以下在 UI 企劃書中容易被遺漏的「底層邏輯」或「隱藏功能」：

### 1. 滑點 (Slippage) 與流動性限制假設
* **滑點定義**：文件中定義了「交易費用」(手續費)，但沒有定義**「滑點」(Slippage)**。在真實市場，條件觸發價不等於真實買到的價格。建議隱藏設定中應加入預設的滑點百分比（例如 0.1%）。
* **成交量限制**：如果策略選到流動性極差的仙股 (Penny stock)，而初始資金高達 $1,000,000，回測若預設「無限吞吐量」會產生不真實的暴利。系統底層是否限制「單筆買入量不可超過該 Bar 成交量的 N%」？

**Codex答/點評：**
1. 這是必要功能，建議預設啟用保守值：滑點 0.05%~0.1%。
2. 建議新增兩個容量限制參數：`max_fill_pct_of_bar_volume`、`min_dollar_volume_filter`。
3. 若成交量不足，建議採「部分成交」而非直接滿額成交，否則績效會系統性偏樂觀。

### 2. 除權息與股票分割 (Corporate Actions)
* **淨值計算**：報表上的「淨值曲線圖」是否考慮了**現金股利 (Dividends)** 的發放並算入帳戶餘額中？
* **價格基準**：進出場機制的「開盤/收盤價」，底層必須嚴格規定是使用「還原權息的價格 (Adjusted Price)」，否則遇到股票一拆十的歷史事件，回測會直接判定為崩盤停損出場。

**Codex答/點評：**
1. 必須明確定義價格模式，建議提供：`Raw`、`Split-adjusted`、`Total-return(adjusted)`。
2. 若策略含股利再投資，淨值曲線應納入股利現金流；否則需在報告中明示「未含股利」。
3. 若不先定義此邏輯，回測可信度會顯著下降。

### 3. 資金複利與再投資邏輯 (Compounding)
* **單利 vs 複利**：「資金與風險管理模組」如果決定每一次可投入單筆風險為 1%，這個 1% 是算在**「初始資金」**上（單利模式、固定資金投入），還是算在隨時間變動的**「當前帳戶總淨值」**上（複利模式，賺錢部位大、賠錢部位小）？這在 UI 上通常也是一個選項或預設的底層隱藏邏輯。

**Codex答/點評：**
1. 必須做成明確選項，建議預設為「當前淨值（複利）」。
2. 報告需同步顯示資金模式，避免不同模式績效被誤比。
3. 建議加「固定風險上限」與「可用現金上限」雙重約束，避免過度槓桿。

### 4. 回測期末的平倉結算 (Mark-to-Market)
* **結算邏輯**：當回測時間軸走到 `end_date`，但手上還有未平倉的部位時，系統該如何處理？
  * 做法 A：強制以最後一天的收盤價計算平倉 (Force Liquidate)，並算入最終獲利與交易次數。
  * 做法 B：不平倉，僅以當日市值計算到「帳戶總淨值」中，但不列入「交易明細清單」的已實現損益（因為還沒賣）。這點後端運算需明確定義。

**Codex答/點評：**
1. 建議支援 A/B 兩種模式，但預設 A（Force Liquidate）方便比較策略。
2. 若選 B，報表要拆出「已實現損益 / 未實現損益」，避免指標誤讀。
3. 指標如勝率、平均持有期、交易次數需明確說明是否含期末強平單。

### 5. 放空借券費率與保證金 (Shorting / Margin)
* **作空成本**：如果左側勾選「作空」，系統是否會隨著「持有期數」扣除年化的融券/借券利息？
* **保證金機制**：若是支援保證金交易（總持倉大於 100% 資金），是否有維持保證金的追繳 (Margin Call) 或爆倉的強平邏輯？

**Codex答/點評：**
1. 若提供做空，借券費率必須入模擬，否則做空績效會偏高。
2. 若允許保證金，需定義維持率、追繳門檻、強平順序（先平虧損最大或流動性最低）。
3. 建議第一版先關閉槓桿，先上線 cash account，再迭代 margin 模式。

### 6. 參數記憶與策略儲存 (State Persistence)
* **狀態暫存**：使用者花了大量時間設定好複雜的進出場條件與風控參數，如果不小心重新整理網頁（F5），參數是否會消失？建議此頁面應實作 **LocalStorage 的暫存機制**。
* **策略範本**：進階建議仿效選股頁面，提供一個「另存為我的回測策略 (Save as Template)」的功能，以便後續直接載入覆用。

**Codex答/點評：**
1. 這是高優先需求，建議至少先做 LocalStorage 自動草稿保存（每 2~5 秒 debounce）。
2. 儲存格式建議加入 `schema_version`，避免未來欄位升級造成舊草稿讀取失敗。
3. 「另存策略」建議與選股頁相同操作模型（儲存、覆蓋、另存新檔、刪除），減少學習成本。
