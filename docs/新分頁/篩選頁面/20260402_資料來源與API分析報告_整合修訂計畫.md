## 3. 修訂策略
- 將「抓取順序與維度」獨立成一節，明確描述：
  - 現行 scheduler 只排 1d（非 1m/5m/1h/1d 全部）
  - incremental_update 與 progressive_backfill 的執行順序（timeframe -> ticker 組 -> 時間區間）
- 將「請求計數」獨立成一節，定義：
  - 程式中的一次請求 = 一次 yf.download 呼叫（單一 interval + 一組 tickers + 一個 period/start-end）
  - max_daily_downloads 的真正含義與限制
- 將「斷點續跑」寫成「目前可恢復程度 vs 不可恢復部分」
- 將「缺口補填」拆成兩個層次：
  - ensure_data（完整度閾值）
  - gap_scanner（連續時間差 > 閾值）
- 新增「凌晨關機情境下的排程重設方案」
  - 比較方案：啟動即跑 / 每日固定時段 / 優先級佇列
  - 指出最大風險與失敗條件
- 新增「額外便宜資料來源（上網查證）」節，採「可行性分級」：
  - 可作日常增量
  - 可作一次性歷史打底
  - 不建議（成本或覆蓋不符）

## 5. 最大風險與失敗條件
- 風險 A：文件與程式不同步
  - 失敗條件：沿用舊文件敘述（如 scheduler 支援全部 timeframe）導致錯誤決策。
- 風險 B：把 yfinance 2,000 限制當作精確 SLA
  - 失敗條件：以固定上限推導產能，忽略實際封鎖策略波動。
- 風險 C：把 max_daily_downloads 誤解為全域每日總額
  - 失敗條件：錯估更新吞吐與可完成時間。
- 風險 D：凌晨排程在關機時段
  - 失敗條件：任務長期不觸發，歷史回補與缺口補填形同停擺。

## 6. 替代方案（高層）
- 方案 1（保守）：維持 yfinance 為主，排程改為「容器啟動補跑 + 每日固定增量」。
- 方案 2（成本最低可用）：日線維持 yfinance；分K用一次性資料包打底，再以免費來源做增量。
- 方案 3（穩定優先）：付費 API 作主來源，yfinance 作 fallback。
